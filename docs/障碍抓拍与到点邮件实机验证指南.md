# 障碍抓拍与到点邮件实机验证指南

## 1. 本次验证目标

这次实机验证只确认四件事：

1. 后端启动时只探测一次摄像头；进入视觉任务后按需打开并复用会话，任务结束立即释放。
2. 小车在边上确认障碍后，先刹车、倒回起始节点、短前行居中并再次刹车，然后才抓拍。
3. 障碍 JPEG 与同名 JSON 写入 `captures/obstacles/`，后端重启后前端仍能查看。
4. 到达起点、每个途径点和终点时分别异步发送 QQ 邮件，邮件失败不改变行程结果。

障碍不会触发邮件。障碍记录不设上限，也不自动删除；长期使用需要人工关注磁盘空间。

## 2. 代码中的实际时机

障碍距离由 `src/tasks/edge_follow.py` 的连续确认逻辑产生。`src/tasks/grid_navigation.py` 在当前边加入动态封锁后调用 `recover_to_start_node()`；该恢复函数内部已经完成节点确认、短前行居中和刹车。只有恢复函数返回后，导航器才调用 `obstacle_result_fn`。

`src/server/ride_service.py` 接收该回调，调用 `ObstacleRecorder` 抓拍并保存记录。保存完成后生成带 `obstacle_id` 的前端消息，随后 `GridNavigator.navigate()` 才进入下一轮 A* 规划。

恢复失败也会保存一条 `recovery_failed` JSON，但不会抓拍，因为小车没有回到可信静止点。

## 3. 启动前配置

在树莓派终端使用 `python3`，不要使用指向 Python 2.7 的 `python`。

当前实机使用已经确认的 Sanhao Face 稳定 V4L2 路径。启动前先核对它仍能解析：

```bash
readlink -f /dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0
```

然后设置：

```bash
export CAMERA_DEVICE=/dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0
```

不要把可能随重启或重新插拔变化的 `/dev/video0`、`/dev/video1` 写成长期启动配置。

QQ SMTP 继续使用现有环境变量，授权码不要写进仓库：

```bash
export MAIL_SMTP_HOST=smtp.qq.com
export MAIL_SMTP_PORT=465
export MAIL_USERNAME='你的发件 QQ 邮箱'
export MAIL_PASSWORD='你的 SMTP 授权码'
export MAIL_FROM='你的发件 QQ 邮箱'
export MAIL_TO='接收通知的邮箱'
```

还要保留实车后端已有的可信初始姿态：

```bash
export CAR_INITIAL_POSITION=C3
export CAR_INITIAL_HEADING=east
```

启动后端前，停止其它会占用摄像头的推流、人脸识别、二维码或全景工具。当前设计中，运行中的后端是摄像头唯一所有者。

## 4. 最小启动检查

按项目现有方式启动后端，然后检查：

```bash
curl http://127.0.0.1:8000/api/health
curl http://127.0.0.1:8000/api/obstacles
```

成功标准：

- `hardware_ready` 为 `true`。
- 最近一次摄像头探测或读帧成功时，`camera_ready` 为 `true`。
- 最近一次摄像头打开或读帧失败时，后端仍能响应，`camera_ready` 为 `false`，并有明确 `camera_error`。
- 没有历史记录时 `/api/obstacles` 返回 `[]`。

`camera_ready` 是最近一次尝试的结果，不表示空闲期间一直占用或实时监控设备。视觉任务读帧失败后会释放失效句柄，并在本阶段原有超时内重新打开；本次超时后，后续识别场景仍会再次尝试，不需要为了清除软件的“永久不可用”状态而重启后端。

## 5. 障碍实机验证

先选一条短边，例如 `C3 → C4`，确保倒车区域和周围人员安全，并准备随时断电或使用现有停车方式。

期望物理顺序：

```text
C3 → C4 行驶
→ 两次新鲜超声波读数确认障碍
→ 立即刹车
→ 封锁 C3—C4 并保留确认距离
→ 倒车返回 C3
→ 检测到 C3 后短前行居中
→ 再次刹车并保持静止
→ 抓拍并保存 JPEG/JSON
→ 前端出现障碍消息
→ A* 绕开 C3—C4 重新规划
```

成功标准：

- 抓拍发生在小车回到 C3、居中并静止之后，不是在障碍物前急停位置拍摄。
- 前端首页原消息项内出现障碍缩略图和“查看障碍记录”。
- 障碍记录页显示 `C3 → C4`、实际确认距离、恢复点 `C3` 和“已恢复并绕行”。
- 摄像头失败时显示“摄像头拍照失败”默认图，不显示伪造照片。
- 本次 `navigate()` 内重新规划不再经过 C3—C4；下一次独立 `navigate()` 清空动态封锁边是已确认设计。

磁盘核对：

```bash
ls -lh captures/obstacles/
```

正常抓拍应看到同 ID 的 `.jpg` 和 `.json`。抓拍失败只应有 `.json`，其中 `image_filename` 为 `null` 且 `capture_error` 有具体原因。

重启后端后再次打开障碍记录页，原记录仍应存在。

## 6. QQ 邮件验证

创建包含一个途径点的短行程。成功标准：

- 到达起点发送第一封。
- 到达途径点发送第二封。
- 到达终点发送第三封。
- 每封正文都包含通知类型、到达点位、完整路线和到达时间。
- 导航线程不等待 SMTP 返回；小车能继续执行下一段。
- SMTP 失败时前端出现系统消息，但到达终点的行程状态仍是 `arrived`。
- 障碍发生时不发送邮件。

## 7. 结束检查

停止后端时确认电机先进入安全状态，随后摄像头被释放。后端停止后，独立摄像头测试工具应能重新打开同一设备。
