# 固定摄像头视觉接客与障碍识别详细执行计划

> 文档状态：阶段 0 已完成本地实现；新阶段 1 已完成 Windows 本地实现与可用测试，二者均等待树莓派实机验证；障碍视觉阶段未实施
>
> 适用项目：Yahboom 4WD 树莓派智能小车
>
> 核心硬件结论：摄像头云台一转动，USB 摄像头会掉线，因此本方案彻底取消摄像头云台动作
>
> 计划目标：在不改变“视频连续取帧识别”方式的前提下，完成选乘客、人脸核验、确认上车、障碍颜色分类、收费站二维码识别、人工移障后自动继续的完整演示闭环

---

## 1. 计划结论

本次不采用“先把前端全部接好，以后再替换视觉实现”，也不采用“所有视觉模块全部重构完成后才开始接前端”这两种完全串行方案。

合理顺序是：

1. 先删除摄像头云台旧逻辑，确认固定摄像头和现有导航基线没有退化。
2. 先确定后端状态、接口和记录字段，因为前端与视觉任务都依赖这份契约。
3. 把后端唯一摄像头会话扩展为连续取帧能力，让人脸、颜色、二维码共享同一摄像头资源。
4. 先完成“选择乘客 → 到达起点 → 验脸 → 确认上车 → 继续行程”的最短产品闭环。
5. 再完成“停车 → 颜色分类 → 收费站扫码或普通障碍倒车重规划”的障碍闭环。
6. 最后完善障碍记录页面、浏览器刷新恢复和实机参数标定。

这样安排的原因不是保守，而是依赖关系决定的：前端不能在接口字段未确定时盲接；视觉任务也不能绕过后端唯一摄像头所有者自己重新打开设备。公共地基完成后，前后端应围绕一个业务闭环同步推进，而不是长期各自开发。

## 2. 已确认的产品需求

### 2.1 固定摄像头约束

- 取消所有摄像头云台转动、扫描和自动寻找目标逻辑。
- 摄像头保持一个实机标定后的固定角度。
- 人脸、颜色和二维码都使用这个固定视角。
- 识别仍然是摄像头视频连续取帧，不把原有视频识别改成“先拍一张照片，再只识别这张照片”。
- 颜色和二维码可以先后识别，但必须复用同一个已打开的摄像头会话，不能在两步之间关闭、重开摄像头。
- 视觉识别期间小车必须保持停车。
- 超声波舵机 BCM 23 与通用舵机硬件类不属于摄像头云台，继续保留。

### 2.2 模拟接客流程

- 前端先从已登记人脸数据集中读取可选乘客。
- 用户必须从乘客列表选择一人，不允许自由输入，不设置默认乘客。
- 小车先前往起点。
- 到达起点并停车后，自动开始固定视角人脸识别。
- 只有识别结果与本次选择的 `passenger_id` 相同才算成功；识别出其他已登记人员也不允许上车。
- 一次识别有超时，沿用当前人工工具的初始值 20 秒，后续实机调整。
- 识别超时后，小车继续停在起点，前端显示“重新识别”和“取消行程”。
- 点击“重新识别”后，在同一活动行程中再次执行一次完整识别，不创建新行程。
- 识别成功后仍不发车，前端显示识别照片和“确认上车”按钮。
- “确认上车”不限时；只有用户点击后才前往途径点和终点。
- 浏览器刷新后，通过现有 `GET /api/rides/active` 恢复当前活动行程、乘客、识别照片和按钮。
- 后端进程重启后的活动行程恢复本期不做。
- 人脸登记继续使用现有命令行工具，本期不做前端登记页面。

### 2.3 人脸识别记录

- 每次最终识别尝试只保存一张原始 JPEG 和一份同名 JSON。
- 成功时保存“确认匹配的那一帧”。
- 超时时优先保存最有诊断价值的一帧：检测到脸时保存距离最接近阈值的帧；从未检测到脸时保存最后一帧。
- JSON 至少记录：行程、期望乘客、检测到的身份、结果、匹配距离、时间、图片写入错误。
- 图片只保存在树莓派本地 `captures/face_verifications/`，不发邮件、不上传云端、不自动删除。
- 保存图片或 JSON 失败不得把“人脸匹配成功”改成失败，也不得触发小车移动；只记录保存错误。
- 不逐帧保存图片，不为每次身份不匹配生成消息。

### 2.4 障碍分类流程

- 超声波连续确认前方有障碍后，电机立即刹车。
- 电机确认停车后，才开始视觉分类。
- 第一版沿用当前 20 cm 障碍阈值，实机阶段再调整。
- 障碍物正面贴大面积颜色块：
  - 红色表示普通障碍。
  - 蓝色表示收费站。
- 收费站同时贴蓝色色块和黑白二维码，两者并排、正对摄像头，二维码保留白色静区。
- 先从连续视频帧中稳定识别颜色；只有稳定蓝色才进入二维码识别。
- 红色稳定确认后，不执行二维码识别。
- 颜色识别失败、颜色冲突、摄像头失败或超时，都按安全策略视为普通障碍处理，但记录必须明确标明“分类失败”，不能伪装成识别到红色。
- 收费站二维码只接受严格格式 `TOLL:<station_id>`，例如 `TOLL:GATE1`。
- 蓝色后未找到二维码、二维码格式错误或二维码超时，前端提示“收费站二维码识别失败，正在重新规划”，随后按普通障碍处理。

### 2.5 普通障碍与收费站动作

普通障碍：

1. 停车并完成视觉分类。
2. 在内存中保留最终记录帧。
3. 把当前边加入动态封锁集合。
4. 沿原边倒回已知起始节点并完成居中。
5. 停车后写入 JPEG 和 JSON。
6. 重新运行 A* 规划绕行。

有效收费站二维码：

1. 前端提示“已识别收费站 GATE1，即将通过收费站”。
2. 小车继续刹车，等待人工移走障碍物。
3. 不增加“已移走”按钮。
4. 超声波连续确认前方畅通后，小车从当前位置继续完成当前边。
5. 当前边不加入动态封锁集合，也不重新从起始节点执行转向和离开节点动作。
6. 等待畅通初始超时设为 60 秒，实机后再调。
7. 60 秒内仍未畅通，则封锁当前边、倒回起始节点并重新规划。

取消规则：

- 在起点等待重试或确认上车时取消：小车已经位于可信节点，立即取消。
- 正常行驶中取消：保持现有安全语义，到前方下一个可信节点停车后取消。
- 在边中间进行颜色、二维码识别或等待收费站清空时取消：停止等待，把当前边临时封锁，倒回起始节点后取消；禁止在边中间直接结束并把当前位置当作节点。

### 2.6 前端范围

- 不显示实时视频。
- 不新增独立“视觉识别”页面。
- 首页叫车表单增加乘客下拉选择。
- 首页消息区展示人脸、颜色、二维码、收费站等待和处理结果。
- 人脸识别成功消息展示缩略图和原图链接。
- 首页按状态显示“重新识别”或“确认上车”按钮。
- 现有障碍记录页面扩展普通障碍/收费站、红/蓝、站点 ID、识别错误、处理结果和恢复状态。
- 自动视觉任务不提供人工启动按钮。

## 3. 当前代码事实与必须修正的问题

### 3.1 已有能力

- `src/hardware/camera.py` 的 `OpenCVCameraSession` 已支持长期打开和 `read_frame()` 连续取帧。
- `src/server/camera_runtime.py` 已保证后端进程只有一个长期摄像头所有者，但目前只向业务层暴露 `capture()` 落盘抓拍。
- `src/algorithms/face_recognition.py` 已有人脸检测、本地数据集加载和 LBPH 风格匹配。
- `src/algorithms/color_detect.py` 已有 HSV 色块识别，当前人工工具默认连续 3 帧同色确认。
- `src/algorithms/qr_detect.py` 已有连续帧二维码解码和 `TYPE:ID` 严格解析基础。
- `src/tasks/edge_follow.py` 已保证超声波只在正常边行进阶段参与阻挡判断，并在确认障碍后刹车。
- `src/tasks/grid_navigation.py` 已支持动态封边、倒车恢复和 A* 重规划。
- `src/server/runtime_state.py` 已有进程内活动行程和事件记录。
- `frontend/app.js` 已在刷新时请求 `GET /api/rides/active`，具备扩展恢复界面的基础。
- `src/server/obstacle_store.py` 已用同名 JSON/JPEG 做本地持久化，适合扩展字段而不是引入数据库。

### 3.2 当前问题

1. `RideService.run_hardware_ride()` 到达起点后只记录“请上车”，随即继续前往后续站点，没有等待人脸核验和确认上车。
2. `RideCreateRequest` 只允许 `start`、`waypoints`、`end`，没有乘客字段。
3. `BackendCamera` 没有向任务层暴露连续帧读取，所以不能安全复用后端唯一摄像头做视频识别。
4. 当前障碍一经确认，`GridNavigator` 立即封锁边并调用 `recover_to_start_node()`，视觉任务没有插入点。
5. 当前障碍照片在倒回节点后重新拍摄，拍不到障碍出现时的原始画面。
6. 当前 `execute_planned_edge()` 包含“调整朝向 → 离开节点 → 沿边行驶”三步。收费站清空后若再次调用它，会错误地把边中间当成节点。
7. 当前障碍记录字段只有边、距离、恢复结果和图片，不能表达颜色、收费站、二维码和处理结果。
8. 当前摄像头云台代码与已确认的硬件事实冲突，继续保留会误导后续开发和实机操作。

### 3.3 最关键的导航修正

收费站清空后必须“续走当前边”，不能重新执行整条边。计划在 `EdgeFollower` 增加一个有真实业务边界的方法：

```text
resume_planned_edge(target_heading, max_seconds, cancel_requested_fn)
```

它只复用现有 `_travel_edge()`，不执行 `_align_to_heading()` 和 `_leave_node()`。这不是为了未来扩展而加的包装，而是为了表达当前真实状态：“小车已经在边中间，朝向已知，只需继续巡线到下一个节点”。

`GridNavigator` 在障碍点刹车后调用上层注入的障碍处理回调。回调只返回两种导航动作：

- `continue_current_edge`：收费站二维码有效且超声波已连续确认畅通。
- `block_and_recover`：普通障碍、任何识别失败、收费站等待超时或取消。

`GridNavigator` 只理解这两个运动动作，不 import 人脸、颜色、二维码或存储模块。识别结果作为不可变上下文原样传给结果回调，用于后续保存记录。

## 4. 目标架构与责任边界

```text
浏览器
  ├─ 乘客选择 / 路线提交 / 重试 / 确认上车 / 取消
  ├─ 活动行程与事件轮询
  └─ 障碍记录展示
             │ HTTP
             ▼
src/server/app.py
  ├─ 严格校验外部请求
  ├─ 创建唯一硬件栈、唯一摄像头会话、视觉任务
  └─ 注册接口，不承载识别循环和导航流程
             │
             ▼
src/server/ride_service.py
  ├─ 接客状态机
  ├─ 人脸重试/确认上车条件等待
  ├─ 障碍业务消息和动作决策
  └─ 组合导航、视觉任务和记录器
       │                    │
       ▼                    ▼
src/tasks/              src/server/*_store.py
  ├─ face_verification      ├─ 人脸 JPEG/JSON
  ├─ obstacle_visual        └─ 障碍 JPEG/JSON
  ├─ grid_navigation
  └─ edge_follow
       │
       ▼
src/hardware/ + src/algorithms/
  ├─ 唯一摄像头、电机、巡线、超声波
  └─ 人脸、HSV 颜色、二维码纯识别能力
```

### 4.1 信任边界

- 浏览器请求、二维码文本、磁盘数据集目录和 JSON 文件都属于不可信输入，必须严格校验。
- `passenger_id` 只接受当前成功加载的人脸标签之一，不兼容别名或多套字段。
- 二维码必须由 `parse_qr_payload()` 校验且 `qr_type == "TOLL"`，不接受大小写修正、空格清理或其他前缀兜底。
- 已通过 schema 的行程对象、已构造的视觉结果和导航动作属于进程内可信契约，不在每层重复校验。

### 4.2 资源所有权

- `BackendCamera` 是后端唯一摄像头所有者；任何自动视觉任务不得创建 `OpenCVCameraSession`。
- `GridNavigationHardware` 继续统一拥有电机、巡线、超声波、蜂鸣器和倒车雷达。
- `RideService` 只编排，不调用 GPIO，不关闭硬件。
- 最外层 lifespan 负责服务退出时唤醒等待中的行程、紧急停车并关闭资源。
- 人脸、颜色、二维码自动任务在同一个行程工作线程中串行执行，因此不需要摄像头锁竞争或任务队列。

## 5. 固定接口与数据契约

### 5.1 乘客列表

```http
GET /api/passengers
```

成功响应采用最简单的字符串数组：

```json
[
  "Alice",
  "张三"
]
```

规则：

- 数组内容来自 `captures/faces/<乘客名>/` 中被识别器实际成功加载的标签，不只看空目录名。
- 标签同时作为第一版的 ID 和界面显示名，不增加头像、别名或数据库主键。
- 第一版在后端启动时加载一次数据集；命令行新增或更换登记照片后，需要重启后端才会更新乘客列表和识别器，不实现运行中热重载。
- 没有有效登记人员时返回空数组；前端禁用叫车按钮并明确提示“没有可用乘客，请先在树莓派命令行登记人脸”。

### 5.2 创建行程

```http
POST /api/rides
Content-Type: application/json

{
  "passenger_id": "Alice",
  "start": "A1",
  "waypoints": ["C2"],
  "end": "E5"
}
```

- 只允许以上四个字段。
- `passenger_id` 必填，必须与当前已加载标签完全相等。
- 无可用摄像头或人脸识别器未就绪时拒绝创建行程，返回明确错误；不创建一个注定无法接客的活动行程。

### 5.3 人脸重试与确认上车

```http
POST /api/rides/{ride_id}/face-verification/retry
POST /api/rides/{ride_id}/confirm-boarding
```

- 两个请求都不需要请求体。
- `retry` 只允许在 `waiting_passenger_retry` 调用。
- `confirm-boarding` 只允许在 `awaiting_boarding_confirmation` 调用。
- 重复点击、错误行程或错误状态返回 409，不静默成功。
- 接口只向正在等待的同一行程工作线程发送命令并唤醒它，不启动第二条导航线程。

### 5.4 行程响应新增字段

`RideStatusResponse` 增加：

```json
{
  "passenger_id": "Alice",
  "face_verification_id": "face_20260714_...",
  "face_verification_image_url": "/api/face-verifications/face_20260714_.../image"
}
```

- 未生成记录或没有成功写入图片时，对应值为 `null`。
- 活动行程接口返回这些字段，浏览器刷新即可恢复乘客和照片。

### 5.5 人脸图片接口

```http
GET /api/face-verifications/{record_id}/image
```

- 只返回 JSON 已登记且实际存在的 JPEG。
- `record_id` 必须通过固定正则校验，禁止路径穿越。
- 本期不增加人脸记录列表页面或删除接口。

### 5.6 障碍记录响应

在现有字段上扩展为：

```json
{
  "id": "obstacle_...",
  "ride_id": "ride-...",
  "created_at": "2026-07-14T10:00:00+00:00",
  "from_point": "A1",
  "to_point": "A2",
  "distance_cm": 16.8,
  "obstacle_type": "toll",
  "detected_color": "blue",
  "classification_status": "success",
  "station_id": "GATE1",
  "recognition_error": null,
  "handling_result": "continued_current_edge",
  "recovery_status": null,
  "recovered_point": null,
  "image_url": "/api/obstacles/obstacle_.../image",
  "capture_error": null
}
```

固定枚举：

- `obstacle_type`：`ordinary`、`toll`。
- `detected_color`：`red`、`blue`、`null`。
- `classification_status`：`success`、`failed`。
- `handling_result`：`continued_current_edge`、`blocked_and_replanned`、`canceled_after_recovery`、`recovery_failed`。
- `recovery_status`：`recovered`、`recovery_failed`、`null`。

识别失败时，安全动作仍按普通障碍执行，但保存为：

```json
{
  "obstacle_type": "ordinary",
  "detected_color": null,
  "classification_status": "failed",
  "recognition_error": "color_timeout",
  "handling_result": "blocked_and_replanned"
}
```

这样既能保证安全，又不会把“没认出来”伪造成“认出了红色”。

## 6. 行程状态机

公开状态只保留下列业务状态，不保留含义重复的 `arrived_pickup`：

| 状态 | 含义 | 允许的用户操作 |
| --- | --- | --- |
| `to_pickup` | 前往乘客起点 | 取消 |
| `verifying_passenger` | 起点停车，正在验脸 | 取消 |
| `waiting_passenger_retry` | 本次验脸未通过 | 重新识别、取消 |
| `awaiting_boarding_confirmation` | 验脸成功，等待确认上车 | 确认上车、取消 |
| `in_trip` | 已确认上车，前往途径点或终点 | 取消 |
| `classifying_obstacle` | 边中停车，正在识别颜色 | 取消并安全倒回 |
| `scanning_toll_qr` | 已确认蓝色，正在扫描二维码 | 取消并安全倒回 |
| `waiting_toll_clearance` | 二维码有效，等待超声波确认畅通 | 取消并安全倒回 |
| `canceling` | 正在到可信节点或倒回起始节点 | 无 |
| `arrived` | 已到终点 | 无 |
| `failed` | 硬件或导航失败 | 无 |
| `canceled` | 已安全取消 | 无 |

障碍状态可以在 `to_pickup` 或 `in_trip` 中发生。处理完成后恢复进入障碍前的行驶阶段；这个“恢复状态”保存在当前导航调用的局部上下文中，不为它新增一套持久状态字段。

接客主流程：

```text
to_pickup
  → verifying_passenger
      → 匹配成功 → awaiting_boarding_confirmation
      → 超时/失败 → waiting_passenger_retry
                       → 重新识别 → verifying_passenger
  → 确认上车 → in_trip
  → arrived
```

障碍子流程：

```text
to_pickup / in_trip
  → 超声波连续确认 + 刹车
  → classifying_obstacle
      → 红色 → 封边、倒回、记录、重规划
      → 失败 → 按普通障碍封边、倒回、记录、重规划
      → 蓝色 → scanning_toll_qr
          → 无效/超时 → 封边、倒回、记录、重规划
          → TOLL:<id> → waiting_toll_clearance
              → 连续畅通 → 记录、续走当前边
              → 60 秒超时 → 封边、倒回、记录、重规划
              → 取消 → 封边、倒回、记录、canceled
```

2. 未移障前持续停车。
3. 人工移走后超声波连续确认畅通。
4. 小车从边中继续，不重复转向或离开节点。
5. 到达下一节点后可信位置正确。

### 第五次：失败与取消

- 蓝色无二维码。
- 非法二维码。
- 颜色模糊。
- 收费站 60 秒未移走。
- 识别过程中摄像头掉线。
- 收费站等待中取消。
- 倒车恢复失败。
- 图片目录不可写。

每个失败用例都必须核对：电机是否安全、可信位置是否真实、状态是否可解释、是否只生成一条最终记录。

## 9. 测试矩阵与成功标准

### 9.1 Windows 本地自动测试

- schema 严格字段与错误码。
- 乘客标签加载与未知乘客拒绝。
- 人脸期望身份、其他身份、无脸、超时、保存失败。
- 颜色红/蓝稳定帧、交替颜色、超时。
- 二维码有效/无效/错误类型/超时。
- 状态机重试、确认、刷新返回字段、各状态取消。
- 导航续走当前边、封边倒车、恢复失败。
- 障碍新 JSON 契约和图片安全读取。
- 单摄像头会话生命周期。

算法和业务测试使用假帧、假识别器、假电机、假超声波缓存和注入时钟，不为测试增加生产专用字段或测试分支。

### 9.2 树莓派实机测试

本地测试不能证明以下事实，必须实机验证：

- USB 摄像头长期稳定性。
- 固定安装角度的可见范围。
- 电机刹车后画面是否仍抖动。
- 20 cm 停车距离是否给颜色块和二维码足够画面面积。
- 人工移障时超声波连续畅通读数是否稳定。
- 边中续走是否能继续抓线并正确到达节点。

### 9.3 最终可验收标准

一次完整演示必须同时满足：

1. 前端只能选择已登记乘客。
2. 小车到起点后停车并只接受指定乘客。
3. 未确认上车前绝不发车。
4. 行驶遇障碍后先停车再视觉识别。
5. 红色障碍倒回、封边并绕行。
6. 蓝色收费站只接受 `TOLL:<id>`。
7. 有效收费站移走后自动续走当前边。
8. 任意视觉失败都选择安全倒回，不盲目前进。
9. 人脸和每个障碍都最多保存一张最终原始照片与同名 JSON。
10. 浏览器刷新能恢复当前进程中的活动行程操作。
11. 全程没有摄像头云台动作，摄像头不因舵机转动掉线。

“页面显示成功”“接口返回 200”或“程序能跑起来”均不构成单独验收通过。

## 10. 推荐提交顺序

每个提交都应单独通过相关测试，提交信息使用 Conventional Commits：

1. `refactor(camera): 删除摄像头云台扫描并保留固定视角工具`
2. `feat(camera): 为后端唯一摄像头增加连续帧读写能力`
3. `feat(ride): 完成指定乘客人脸核验与确认上车闭环`
4. `docs(vision): 更新固定摄像头接客阶段与实机验证说明`
5. `refactor(navigation): 增加障碍决策插入点与当前边续走能力`
6. `feat(vision): 接入障碍颜色分类与收费站二维码识别`
7. `feat(frontend): 展示收费站障碍处理状态`
8. `docs(test): 记录固定摄像头视觉与收费站实机验收结果`

不要把所有阶段压成一次大提交。任何阶段实机失败时，应能回到上一阶段的已验证闭环。

树莓派上不使用 Git。Windows 仓库完成并提交后，只把本阶段实际修改的 `src/`、`frontend/` 和必要文档通过 WinSCP 覆盖到 `/home/pi/4wd_project`，再按阶段执行实机验证。
