# 代码地图模板实施问题记录

本文件按 `docs/interactive-code-map-sop.md` 记录模板能力下沉过程中真实发生、已经验证且可复用的问题。

### 模板示例入口目录无法同时邻接两个目标目录

- 问题现象：模板原顺序为 `tasks → services → storage`，入口位于最左侧，却同时调用中间和最右侧目录。迁移自动路由后，`tasks → storage` 会被正确识别为跨越非相邻目录并明确报错。
- 原因：安全正交路由需要使用目录间留白；从最左目录直接连接最右目录会横穿中间目录正文。旧手工 lane 没有表达这条布局约束。
- 解决方案：把模板示例目录调整为 `services → tasks → storage`，让入口位于中间，两个跨目录目标分别使用左右相邻走廊；同目录的 service 内部调用继续使用左侧与入口之间的走廊。
- 验证结果：7 项通道与目标端口算法测试全部通过，Vite 生产构建成功转换 9 个模块。Chromium 1440×900 下，入口单独展开时 4 条线、同时展开 `load_data()` 时 5 条线，调用 ID、路径、源锚点和目标端口均无重复，线路与非端点方法卡片交集为 0；错误面板为空，控制台 0 错误、0 警告，跳转、Esc 返回、输入输出、Hover 和缩放正常。
- 可复用经验：一个入口目录需要调用左右两个业务目录时，优先把入口放在中间；没有定义跨目录总线时，非相邻目录调用必须通过布局调整解决，不能让线路穿过中间目录。

### 相对命令行无法按工作目录识别预览进程

- 问题现象：浏览器验收结束后，按进程命令行包含 `code-map-template` 的条件关闭预览服务没有匹配到进程，端口 4174 仍处于监听状态。
- 原因：预览进程从模板目录启动，但 Node 命令行只保存了相对入口 `node_modules/vite/bin/vite.js`，没有包含工作目录绝对路径，因此仅检查命令行无法证明它属于哪个地图。
- 解决方案：启动时保存进程 PID；清理时联合核对监听端口、拥有者 PID 和实际 Node 进程，本次确认端口 4174 的拥有者就是启动时返回的 PID 30392 后，只停止该进程。
- 验证结果：停止后重新查询监听状态，端口 4174 已关闭；随后安全清理浏览器快照、截图、`dist` 和 `node_modules`，未影响其他 Node 服务。
- 可复用经验：通过相对脚本启动临时服务时，不要依赖命令行路径做唯一归属判断；应在启动时保存 PID，并在结束时用“预期 PID + 监听端口”双重确认后停止。

### 聚焦按钮的悬停样式覆盖激活样式

- 问题现象：浏览器检查确认“查看出线”已经是 `aria-pressed="true"`，线路也正确聚焦，但点击后鼠标停留在按钮上时，按钮仍显示浅色悬停背景，看不出当前方向已经激活。
- 原因：`.line-focus-button:hover:not(:disabled)` 的 CSS 选择器优先级高于单独的 `[aria-pressed="true"]` 规则，悬停样式覆盖了激活背景。
- 解决方案：让激活规则同时匹配普通激活和激活后悬停两种状态，保证 `aria-pressed="true"` 时始终使用蓝底白字。
- 验证结果：模板和网格导航地图重新生产构建成功；Chromium 中点击后保持鼠标悬停，两个页面的激活按钮均计算为蓝色背景 `rgb(23, 111, 209)`、白色文字 `rgb(255, 255, 255)`，聚焦线路数量分别保持 4 条和 31 条，控制台无错误或警告。
- 可复用经验：可切换按钮必须同时检查“激活”“悬停”和“激活且悬停”三种组合状态；不能只通过 DOM 属性断言判断视觉反馈正确。
