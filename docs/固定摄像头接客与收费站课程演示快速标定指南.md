# 固定摄像头接客与收费站课程演示快速标定指南

## 1. 目标与原则

这次不重新验收人脸、颜色、二维码和导航模块，只解决它们组合到实车后是否能顺利完成一次课程演示。

测试原则：

- 先把摄像头角度、色块、二维码和障碍位置调到明显可用。
- 从第一次完整流程开始录像；第一次完整成功即可结束。
- 失败时只处理第一个失败点，每次只改一个物理条件或参数。
- 同一问题调整两次仍失败，再退回对应独立工具定位。
- 不做五次重复、成功率统计、压力测试和异常覆盖。

成功标准不是“每个模块又测了一遍”，而是最终视频中能清楚看到：

```text
选择乘客并叫车
→ 到起点停车并通过人脸识别
→ 确认上车后继续
→ 红色普通障碍：拍照、倒回、重新规划
→ 蓝色收费站：识别颜色和二维码、保持停车
→ 人工移走收费站
→ 超声波连续确认畅通后沿当前边继续
→ 到达终点并停车
```

## 2. 开始前一次性准备

### 2.1 上传与清理

使用 WinSCP 覆盖：

```text
D:\4wd_project\src\       -> /home/pi/4wd_project/src/
D:\4wd_project\frontend\  -> /home/pi/4wd_project/frontend/
```

不要上传本地的 `.playwright-cli/`、`output/`、`tmp/` 和 `captures/`，不要在树莓派运行 Git。

新版障碍记录字段与旧版不同。用 WinSCP 把树莓派的 `captures/obstacles/` 重命名为 `captures/obstacles_old/`，再新建空的 `captures/obstacles/`。这比为了旧数据增加兼容代码简单可靠。

### 2.2 固定现场条件

- 使用最终录像时的场地、灯光、电池和障碍物。
- 摄像头不转动云台，只允许人工微调固定安装角度；调好后拧紧。
- 红色色块贴在普通障碍正面。
- 蓝色色块和二维码贴在收费站正面，二者互不遮挡。
- 二维码固定为 `TOLL:GATE1`，四周保留完整白边。
- 每次重新启动后端前，把小车放回真实节点中心，并设置正确车头朝向。

带电机测试时，一人看车，一人看后端终端。异常先按 `Ctrl+C`；程序未及时停车时立即断电。小车停在两节点之间后，禁止直接开始新行程，必须先人工放回可信节点。

### 2.3 核对摄像头

树莓派执行：

```bash
cd /home/pi/4wd_project
sudo systemctl stop 4wd-camera.service
pgrep -af 'stream_camera_mjpeg|uvicorn|test_.*(photo|face|color|qr)'
sudo fuser -v /dev/video*
readlink -f /dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0
```

稳定路径必须能解析到真实 `/dev/videoN`，并且不能有其它程序占用摄像头。路径不存在时停止测试，不要改用不稳定的 `/dev/video0` 猜测。

## 3. 默认基线：先不要改参数

当前默认值已经适合作为第一次实机基线：

| 项目 | 默认值 | 第一次怎么做 |
| --- | ---: | --- |
| 摄像头分辨率 | 640×480 | 保持 |
| 颜色最小面积 | 1500 px | 保持 |
| 颜色连续确认 | 2 帧 | 保持 |
| 颜色超时 | 15 秒 | 保持 |
| 二维码超时 | 15 秒 | 保持 |
| 障碍距离 | 20 cm | 保持 |
| 收费站畅通确认 | 3 次新鲜读数 | 保持 |
| 收费站移障等待 | 60 秒 | 保持 |
| 单边最长时间 | 20 秒 | 保持 |
| 倒车恢复最长时间 | 8 秒 | 保持 |

结合后续实机出现的 `0.327～0.329` 正确乘客距离，课程演示使用阈值 `0.36`、连续 `1` 帧、最长 `20` 秒。该设置优先提高一次演示通过率，单帧偶然误认也会直接成功，因此录制前必须让另一名已登记乘客做一次交叉检查。

需要知道一个现有问题：障碍距离暂时有两个代码来源。

- 导航停车：`src/server/hardware_factory.py` 中的 `ultrasonic_threshold_cm=20`。
- 收费站放行：`src/config.py` 中的 `ULTRASONIC_THRESHOLD = 20`。

默认值相同时没有问题。若后面确实要调整距离，两个位置必须同步修改；最终录制前不临时重构这个技术债。

## 4. 五分钟静态标定

这里只检查最终物理摆放，不让小车运动，也不重测全部算法。

把障碍物放在摄像头前约 20 cm 的位置。目标是让固定角度同时满足：乘客正脸完整、色块不被裁掉、二维码足够大。

### 4.1 红色色块

确保后端和其它摄像头程序已停止，只运行一次：

```bash
python3 src/tools/test_color_detect.py \
  --device-path /dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0 \
  --colors red,blue \
  --stable-frames 2 \
  --min-area 1500 \
  --timeout 15 \
  --output captures/calibration_red.jpg
```

显示红色、较快完成连续 2 帧确认，而且标注区域未被画面边缘裁掉，就直接通过。当前红纸实机图像的饱和度中位数约为 109，红色 HSV 饱和度下限已从 120 放宽为 100；面积仍保持 1500，避免把背景小色块当成障碍标记。

### 4.2 蓝色收费站和二维码

按最终方式同时贴好蓝色色块与二维码，只运行一次：

```bash
python3 src/tools/test_qr_detect.py \
  --device-path /dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0 \
  --width 640 \
  --height 480 \
  --timeout 15 \
  --save-success-photo
```

期望输出：

```text
Valid QR code detected.
Raw text: TOLL:GATE1
Type: TOLL
Identifier: GATE1
```

成功照片中二维码应正对摄像头、四周白边完整，宽度大致占画面的三分之一。一次能较快解码就结束，不再换尺寸反复测。

### 4.3 只有改变摄像头角度才补看乘客画面

如果为了障碍物调整过摄像头，用普通拍照工具拍一张已登记乘客在最终上车位置的照片，只确认正脸完整入镜。不重新采集人脸、不重新统计识别距离。

## 5. 启动最终后端

停止独立摄像头工具，把小车放回真实可信节点。以下 `C3` 和 `east` 只是示例，必须换成现场真实位置和朝向：

```bash
cd /home/pi/4wd_project

export CAMERA_DEVICE=/dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0
export CAR_INITIAL_POSITION=C3
export CAR_INITIAL_HEADING=east

python3 -m uvicorn src.server.app:app \
  --host 0.0.0.0 \
  --port 8000 \
  --workers 1
```

不要使用 `--reload` 或多个 worker，否则可能重复占用 GPIO 和摄像头。

另开 SSH 终端：

```bash
curl http://127.0.0.1:8000/api/health
curl http://127.0.0.1:8000/api/passengers
```

开始录像前只确认四件事：

- `hardware_ready` 和 `camera_ready` 都是 `true`。
- 乘客列表包含演示乘客。
- 前端没有显示“后端不可用”。
- 页面当前位置与小车真实节点一致。

## 6. 直接开始完整录像

静态标定通过后，第一次完整流程就是正式候选录像。

- 一次完整成功：保留视频，停止测试。
- 失败：记录后端最后一段日志、页面状态和障碍照片，只处理第一个失败点。
- 调整后：小车放回可信节点，重启后端，从头录下一次。

完整流程中只关注这些结果：

| 观察点 | 正常结果 |
| --- | --- |
| 到达起点 | 刹车后才验脸，确认上车前不移动 |
| 红色普通障碍 | 识别红色并留图，倒回可信节点后重新规划 |
| 蓝色收费站 | 识别蓝色和 `TOLL:GATE1`，障碍未移走时保持停车 |
| 人工移走收费站 | 提示“即将通过收费站”，随后沿当前边继续 |
| 边中续走 | 不重新转向，能到达下一可信节点 |
| 最终到达 | 实车停车位置与页面位置一致 |

## 7. 失败后只按现象调整一项

### 7.1 颜色超时或误判

先看保存的障碍照片：

| 看到的现象 | 只做这一项 |
| --- | --- |
| 色块太小 | 增大实体色块 |
| 色块被裁掉 | 微调摄像头固定角度或障碍高度 |
| 反光、颜色发白 | 调整灯光或改用哑光色纸 |
| 红蓝目标同时入镜 | 移走无关色块 |
| 色块清晰且面积足够，仍被过滤 | `OBSTACLE_COLOR_MIN_AREA` 从 1500 改为 1200 |
| 背景小物体被误认 | 增大目标，并把最小面积改为 2000 |

不要把连续确认帧数直接降到 1。改一次仍失败，再运行第 4.1 节工具查看真实面积和标注图。

### 7.2 蓝色成功，但二维码失败

按顺序一次只改一项：

1. 确认内容严格为 `TOLL:GATE1`，没有中文冒号、空格或小写字母。
2. 恢复二维码四周白色静区。
3. 减少倾斜、弯曲和反光。
4. 增大打印尺寸，使二维码约占画面宽度三分之一。

仍失败再运行第 4.2 节工具。诊断含义：

- `no_qr_corners`：通常太小、被裁切或白边不完整。
- `corners_detected_but_not_decoded`：通常模糊、倾斜、反光或打印质量差。
- `decoded_payload_invalid`：二维码内容不符合格式。

只有同一位置能够成功、但总在 15 秒边缘才完成，才把 `TOLL_QR_TIMEOUT_SECONDS` 改为 20 秒。增加时间不能修复看不清的二维码。

### 7.3 停车位置导致目标看不全

先增大标识或微调固定摄像头。只有固定角度已经兼顾乘客和障碍，才调整距离：

- 停得太近、目标被裁掉：两个 20 cm 同时改为 22 cm。
- 停得较远、目标过小：优先增大标识；仍不合适再同时改为 18 cm。

每次只改 2 cm。降低阈值会缩短安全距离；有碰撞可能时立即恢复 20 cm。修改后上传 `src/config.py` 和 `src/server/hardware_factory.py`，并重启后端。

### 7.4 障碍未移走却提前续走

这是安全问题，立即停止，不要降低阈值或确认次数。关闭后端后运行：

```bash
python3 -m src.tools.test_ultrasonic --monitor --threshold 20
```

障碍在场时必须显示 `OBSTACLE`。若显示 `safe`，调整障碍宽度、正对方向或超声波探头方向，让探头确实打到障碍物。

### 7.5 移走收费站后仍不继续

用同一超声波命令确认移障后能稳定显示 `safe`：

- 始终显示 `OBSTACLE`：移开前方墙壁、桌腿等物体。
- 在两种状态间跳变：先调整场地和探头，不改软件。
- 稳定显示 `safe`，后端仍不续走：保存日志，这是程序问题，不是调参问题。

只有读数稳定、三次确认又确实影响演示节奏，才把 `TOLL_CLEARANCE_CONFIRM_SAMPLES` 从 3 改为 2。

### 7.6 续走丢线或普通障碍倒车失败

续走丢线先检查人工移障时是否碰车、黑线是否完整露出、小车是否已经偏离黑线。这不是视觉参数问题。

倒车动作正确、已经接近原节点，只是 8 秒到期时，才把倒车恢复上限从 8 改为 10 秒。方向错误或完全找不到线时，增加超时没有意义，应停止录像并按导航问题排查。

如果创建行程返回 `internal_error`，查看 Uvicorn 日志。出现 `obstacle_processing_not_ready` 说明障碍视觉或收费站超声波任务未初始化，不要通过反复提交行程试探。

## 8. 修改参数后怎么重新开始

参数都在 Windows 仓库修改，树莓派只接收 WinSCP 覆盖文件：

1. 只修改当前故障对应的一项；障碍距离例外，需要同步两个来源。
2. 覆盖对应文件后，在 Uvicorn 终端按 `Ctrl+C`。
3. 把小车放回可信节点，重新设置位置、朝向和摄像头路径。
4. 重启后端，从头开始下一次录像。
5. 调整后更差就恢复默认值，不叠加第二项修改。

简单记录这些信息就够了：

```text
路线和初始朝向：
色块、二维码尺寸和摄像头角度：
第一次失败点：
只修改了什么：
修改后的结果：
最终视频是否完整成功：
实车终点与页面位置是否一致：
```

## 9. 什么时候结束

一段视频完整展示以下结果，就停止测试：

1. 指定乘客通过，确认上车前小车不动。
2. 红色普通障碍识别后倒回并重新规划。
3. 蓝色收费站读出 `TOLL:GATE1`，移走前始终停车。
4. 移走收费站后沿当前边继续。
5. 前端照片、颜色、站点 ID 和处理结果与现场一致。
6. 小车最终真实位置与页面一致并停车。

不再为了参数更漂亮、响应再快一点或覆盖更多异常分支继续增加风险。
