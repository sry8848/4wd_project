# Yahboom 4WD 智能小车人脸识别实现报告

## 摘要

本项目在 Yahboom 4WD 树莓派智能小车上实现了一套离线人脸识别系统。它的核心做法是：录入时把照片保存到磁盘；命令行识别或接客后端启动时，再从每张有效照片中提取一份人脸特征并保留在内存中。识别时把摄像头中的人脸与全部已登记特征逐一比较，找到距离最近的样本，再通过距离阈值和连续帧确认决定是否通过。默认录入 10 张，并不是把 10 张照片合成一张“平均脸”，而是建立 10 份独立参考特征。

系统先通过 OpenCV Haar 级联分类器定位正面人脸，再使用基于局部二值模式直方图（Local Binary Pattern Histogram，LBPH）思想的本地特征描述方法区分人员。照片、特征计算和识别判断均在树莓派本地完成，不依赖云端接口，也不需要互联网。在此基础上，功能已经接入小车接客业务，实现指定乘客核验、失败重试、确认上车和核验记录保存。

## 核心实现：录入多少张，就使用多少张有效照片

对当前实现来说，“学习一张照片”的准确含义是：程序从照片中检测出一张主要人脸，将该人脸转换成一份 LBPH 特征描述，并把这份描述作为以后识别时的独立参考。它不是神经网络训练，也不会把多张照片合并成一个模型参数。

录入数量与实际使用数量的关系如下：

| 录入情况 | 实际建立的参考特征 |
| --- | ---: |
| Alice 录入 10 张，10 张都能检测到人脸 | Alice 有 10 份特征 |
| Alice 录入 10 张，其中 2 张损坏或检测不到人脸 | Alice 有 8 份特征，另外 2 张被跳过 |
| Alice 原有 10 份有效特征，又追加 5 张有效照片 | Alice 有 15 份特征 |
| Alice 10 张、张三 10 张，全部有效 | 系统共比较 20 份特征 |

识别一帧新画面时，程序执行以下步骤：

```text
摄像头当前人脸
    ↓ 提取 1 份 LBPH 特征
分别与 Alice_1、Alice_2……张三_10 比较
    ↓
找到距离最小的那一张参考照片
    ↓
距离 ≤ 0.31：得到该照片所属人员的候选标签
距离 > 0.31：判定为未知人员
    ↓
指定乘客连续匹配 2 帧：最终核验成功
```

默认选择 10 张，是为了用较少样本覆盖本人正常的表情、轻微角度和光照变化，同时控制录入时间与树莓派逐张比较的计算量。照片不是越多越好：模糊照片、过暗照片、极端角度照片，或者误放入其他人员照片，都会成为独立参考，反而可能降低识别可靠性。当前课程演示优先使用 10 张清晰、单人、以正脸为主且略有变化的照片；只有在另一种正常现场光照下经常无法识别时，才追加少量同一人的合格照片。

命令行识别工具每次启动都会重新读取全部照片。接客后端则在启动时一次性建立特征库，因此新增或替换录入照片后，需要重启后端，网页乘客列表和自动核验才会使用最新样本。

## 1. 项目目标

本功能要解决的不是通用身份认证问题，而是课程智能小车中的“指定乘客核验”问题。系统需要满足以下要求：

1. 可以使用小车摄像头录入一个或多个人员。
2. 可以在视频画面中检测人脸，并判断是否属于已登记人员。
3. 识别到其他已登记人员或陌生人时，不能把其误当成本次指定乘客。
4. 到达接客点后必须先停车验脸，识别成功也不能自动发车。
5. 人脸识别失败后允许在同一行程内重新识别。
6. 人脸数据和核验记录保存在本地，不上传云端。
7. 摄像头异常或程序退出后必须释放设备，不能影响后续拍照和识别。

## 3. 系统结构

人脸识别按职责拆分为五层：

| 层次 | 主要职责 | 关键实现 |
| --- | --- | --- |
| 摄像头层 | 打开摄像头、连续读取视频帧、保存 JPEG、释放设备 | `src/hardware/camera.py`、`src/server/camera_runtime.py` |
| 算法层 | Haar 人脸检测、LBPH 特征计算、距离比较和陌生人拒绝 | `src/algorithms/face_recognition.py` |
| 操作工具层 | 人脸录入和独立识别测试 | `src/tools/test_face_recognition.py` |
| 业务层 | 核验本次指定乘客、处理成功、超时和取消 | `src/tasks/face_verification.py`、`src/server/ride_service.py` |
| 展示与接口层 | 提供乘客列表、创建行程、重新识别、确认上车和照片访问 | `src/server/app.py`、`frontend/` |

完整数据流如下：

```text
命令行录入人员
    ↓
captures/faces/<人员标签>/ 保存样本
    ↓ 每张有效照片独立提取一份特征
后端启动时建立“人员标签 → 多份参考特征”的本地特征库
    ↓
前端读取人员列表并选择本次乘客
    ↓
小车到达起点并停车
    ↓
摄像头连续取帧，核验是否为指定乘客
    ├─ 失败或超时：保持停车 → 重新识别或取消行程
    └─ 连续匹配成功：保存记录 → 等待确认上车 → 继续行程
```

## 4. 人脸录入实现

### 4.1 数据组织方式

每名人员对应一个独立目录，目录名就是系统使用的人员标签：

```text
captures/faces/
├── Alice/
│   ├── sample_*.jpg
│   └── ...
└── 张三/
    ├── sample_*.jpg
    └── ...
```

人员标签只允许 1 至 32 个中英文字符、数字、下划线或连字符，禁止空格、斜杠和任意路径字符，从输入边界避免错误目录和路径注入问题。

### 4.2 录入过程

录入工具从摄像头连续读取画面。只有当前帧中恰好检测到一张正面人脸时，才接受并保存该帧。默认录入 10 张照片，相邻样本至少间隔 0.5 秒，避免短时间保存大量几乎完全相同的画面。录入阶段保存的是完整画面；后续建立特征库时，程序会在每张画面中重新检测最大人脸，只从人脸区域提取特征。

实际录入命令示例：

```bash
cd /home/pi/4wd_project

python3 src/tools/test_face_recognition.py enroll Alice \
  --device-path /dev/v4l/by-id/usb-lihappe8_Corp._Sanhao_Face-video-index0 \
  --count 10 \
  --timeout 20 \
  --cascade /usr/share/opencv/haarcascades/haarcascade_frontalface_default.xml
```

要求画面中只能有一名人员。录入过程中人员以正视摄像头为主，并轻微改变表情和头部角度，使样本能够覆盖现场正常变化。程序不会自动覆盖同名人员的旧照片；再次录入会向同一目录追加样本。追加后，参考特征数量也会随有效照片数量增加。

## 5. 人脸识别算法

### 5.1 Haar 人脸检测

Haar 分类器负责回答“画面中的人脸在哪里”，不负责判断人是谁。每一帧进入算法后执行以下处理：

1. 将彩色图像转换为灰度图。
2. 使用直方图均衡化改善室内光照不均造成的对比度问题。
3. 使用 Haar 级联分类器检测正面人脸。
4. 过滤宽或高小于 60 像素的人脸框。
5. 按人脸面积从大到小排序，优先处理画面中的主要人脸。

当前检测参数为：缩放因子 `1.1`、相邻区域确认数 `5`、最小人脸尺寸 `60×60` 像素。这组参数在检测能力和树莓派计算负担之间取得了适合当前演示的平衡。

### 5.2 LBPH 特征描述

检测到人脸后，系统将人脸区域缩放为 `96×96` 像素，并再次执行灰度化和直方图均衡化。随后使用局部二值模式描述纹理：

1. 对每个像素比较周围 8 个邻居与中心像素的亮度。
2. 大于等于中心值记为 `1`，否则记为 `0`，得到一个 8 位局部纹理编码。
3. 将人脸划分为 `8×8` 个小区域。
4. 为每个区域统计 256 种纹理编码的归一化直方图。
5. 将所有区域直方图连接为整张人脸的特征描述子。

这种方法不需要 GPU，也不依赖 OpenCV 扩展模块 `cv2.face`，只使用当前树莓派已经安装的 OpenCV 和 NumPy，适合资源有限、人员数量较少的课程演示环境。

### 5.3 身份匹配与陌生人拒绝

后端启动时读取各人员目录中的照片，对每张可用照片重新检测最大人脸，并分别建立特征描述。无法读取或没有检测到人脸的照片会被跳过，不计入实际参考数量。如果某个人员目录最终没有一张可用照片，该人员不会进入可选乘客列表。

识别实时画面时，系统计算当前人脸与所有登记样本之间的卡方距离。距离越小，表示局部纹理越相似。当前实现不先求某个人的平均特征，而是保留每张照片的独立特征，并执行逐张最近邻比较。

系统采用最近样本匹配：

- 找到距离最小的登记样本及其人员标签。
- 最小距离小于或等于阈值时，才接受为已登记人员。
- 距离超过阈值时返回“未知人员”，不会强制分配一个姓名。

因此，如果 Alice 有 10 张有效照片、张三有 10 张有效照片，那么识别当前一张人脸时会得到 20 个距离，再从中选择最小值。增加样本能覆盖更多正常外观变化，但也会线性增加比较次数；更重要的是，错误样本也会参与最近邻竞争，所以样本质量和标签正确性比单纯增加数量更重要。

当前实机中，正确乘客的距离主要集中在 `0.266～0.299`。据此将识别阈值设为 `0.31`，既保留少量现场波动余量，又避免阈值过宽造成误识别。

### 5.4 连续帧确认

单帧可能受到眨眼、运动模糊或瞬时光照影响，因此业务核验不以单帧结果作为最终结论。当前系统要求指定乘客连续匹配 `2` 帧才判定成功。

如果中间出现以下任一种情况，连续计数立即清零：

- 没有检测到人脸；
- 检测到陌生人；
- 识别到其他已登记人员；
- 当前距离超过阈值。

该规则降低了偶然单帧命中直接放行的风险。

## 6. 接客业务集成

### 6.1 后端启动

后端启动时完成以下工作：

1. 通过稳定路径探测 USB 摄像头是否可用。
2. 加载系统 Haar XML 文件。
3. 读取 `captures/faces/` 下的登记样本。
4. 建立本地人脸特征库。
5. 将成功加载的人员标签作为可选乘客列表。

如果没有有效样本，后端不会伪造可选人员；创建接客行程时会明确返回“人脸识别器没有可用登记样本”。

### 6.2 前后端接口

| 接口 | 作用 |
| --- | --- |
| `GET /api/passengers` | 返回后端已成功加载的人员标签 |
| `POST /api/rides` | 创建包含 `passenger_id` 的接客行程 |
| `GET /api/rides/active` | 页面刷新后恢复当前活动行程状态 |
| `POST /api/rides/{id}/face-verification/retry` | 在同一行程中重新执行人脸核验 |
| `POST /api/rides/{id}/confirm-boarding` | 确认乘客已经上车并允许继续导航 |
| `GET /api/face-verifications/{record_id}/image` | 读取合法核验记录对应的照片 |

创建行程时，后端会校验 `passenger_id` 是否属于当前已经加载的人员列表，不能由前端随意提交一个不存在的姓名。

### 6.3 到达起点后的状态控制

小车到达起点后先停车，再进入指定乘客核验流程。系统使用三个明确状态表示当前阶段：

| 状态 | 含义 | 小车行为 |
| --- | --- | --- |
| `verifying_passenger` | 正在连续取帧核验指定乘客 | 保持停车 |
| `waiting_passenger_retry` | 本次 20 秒核验未通过 | 保持停车，等待重试或取消 |
| `awaiting_boarding_confirmation` | 指定乘客已经识别成功 | 继续停车，等待确认上车 |

识别成功只代表“摄像头前人员与本次指定乘客匹配”，不代表乘客已经实际坐稳。因此系统不会自动发车，只有前端点击“确认上车”后才继续前往途经点和终点。

## 7. 摄像头资源管理

运行中的后端是摄像头的唯一所有者。普通拍照、人脸识别、颜色识别和二维码识别均复用同一个 `BackendCamera`，避免多个进程或任务同时打开同一 USB 摄像头。

人脸核验期间复用当前摄像头会话连续读取独立视频帧。短暂读帧失败时，系统释放失效句柄，并在本次 20 秒期限内尝试重新打开。无论最终结果是成功、超时、取消还是异常，核验任务都会在 `finally` 中关闭摄像头会话，使后续视觉任务能够继续使用设备。

## 8. 核验记录与隐私处理

人脸登记样本保存在：

```text
captures/faces/<人员标签>/
```

每次完成的成功或超时核验会在以下目录保存一份 JSON 记录，并在存在可用视频帧时保存同名 JPEG：

```text
captures/face_verifications/
```

JSON 记录包含：记录编号、行程编号、期望乘客、实际检测标签、核验结果、匹配距离、创建时间、图片文件名和图片保存错误。JSON 使用临时文件写入后再原子替换，避免中途写入造成半份记录。

系统严格校验记录编号和图片文件名，图片接口只能访问已由合法 JSON 登记且真实存在的 JPEG，不能通过路径参数读取其他文件。

识别判断与照片存储相互隔离：如果照片编码或写盘失败，系统会记录保存错误，但不会把已经确认成功的人脸结果改判为失败，也不会因此错误驱动小车。

所有人脸样本和核验照片只保存在树莓派本地，不调用百度云等外部人脸 API，不需要 API Key，也不通过互联网上传个人信息。

## 10. 关键问题与解决方法

### 10.1 旧系统软件源失效

树莓派使用 Raspbian 10（buster），原清华镜像中的旧发行版目录已经下线，导致 `apt update` 返回 404。调试时改用仍保存 buster 软件包的历史仓库，完成 OpenCV、NumPy 和 Haar 数据文件安装。

### 10.2 OpenCV 未提供默认 Haar 目录

当前 OpenCV 4.1.2 中 `cv2.data.haarcascades` 不可用，因此程序不能依赖自动查找。项目将实机确认的 XML 路径集中配置为：

```text
/usr/share/opencv/haarcascades/haarcascade_frontalface_default.xml
```

### 10.3 摄像头编号会变化

`/dev/video0` 只是当前枚举编号，重启或重新插拔后可能变化。项目改用摄像头按 USB 身份生成的稳定路径，并在使用前通过 `readlink -f` 核对真实设备节点。

### 10.4 误识别控制

系统没有只依赖单次最相似结果，而是组合了四层约束：

1. Haar 必须先检测到真实人脸区域。
2. 匹配距离必须不超过 `0.31`。
3. 必须连续匹配 `2` 帧。
4. 识别标签必须等于本次行程指定乘客。

这四项共同避免“画面里有一张脸就通过”或“识别到任意登记人员都通过”。

## 11. 适用边界

该系统面向固定摄像头、少量已登记人员和受控光照下的课程演示。Haar 正面人脸检测对大角度侧脸、严重逆光、遮挡和过远人脸较敏感；LBPH 纹理特征也会受光照和摄像头画质影响。

系统没有活体检测能力，不能防止照片或屏幕翻拍攻击，因此不应作为门禁、支付或高安全等级身份认证系统。本项目的验收目标是完成智能小车指定乘客核验闭环，而不是替代专业生物识别产品。

## 12. 总结

本项目完成了从人脸样本录入、Haar 人脸检测、LBPH 本地特征识别、陌生人拒绝、连续帧确认，到前端乘客选择、后端指定人员核验、失败重试、确认上车和本地记录保存的完整实现。

与依赖云端接口的方案相比，本实现不需要联网和密钥，数据边界清楚，能够在树莓派现有硬件和旧版软件环境中独立运行。测试结果证明系统能够正确识别指定乘客、拒绝其他人员，并在识别过程中保持小车停车和摄像头资源可恢复，达到了课程项目的人脸识别与接客演示要求。
