# 交互式代码地图生成标准流程

## 1. 目标

从一个真实功能入口出发，生成一张面向新手的交互式代码地图。地图按真实目录和文件归属，展示该入口实际可达的全部项目自有方法、执行步骤和调用关系。

每次任务复制已经验收的模板，不重新设计页面交互：

```text
visualizations/code-map-template/
├─ index.html
├─ package.json
├─ package-lock.json
├─ tests/
│  └─ route-layout.test.js
└─ src/
   ├─ map-data.js
   ├─ main.js
   ├─ call-lines.js
   ├─ route-layout.js
   └─ styles.css
```

模板负责嵌套卡片、输入输出折叠、字段提示、缩放拖动、调用线、方法跳转、箭头反向追溯、Esc 返回和错误显示。新任务主要完成真实代码分析、数据替换、布局调整和浏览器验收。

## 2. 模板使用契约

按下表使用模板，无需每次重新研究全部实现：

| 文件 | 默认操作 |
|---|---|
| `index.html` | 修改页面标题、副标题和画布说明 |
| `src/map-data.js` | 完整替换为本次真实代码数据；示例对象是数据格式参考 |
| `src/styles.css` | 只在真实目录列数、宽度或视觉检查发现布局问题时修改 |
| `src/main.js` | 默认不修改；只有真实目录层级不受支持或用户要求改变交互时才查看和调整 |
| `src/call-lines.js`、`src/route-layout.js` | 默认不修改；模板已提供自动正交路由，只有目录走廊模型不适用或用户要求改变布线行为时才调整 |
| `tests/route-layout.test.js` | 保留并运行；修改路由算法时同步补充直接覆盖真实规则的测试 |
| `package.json`、`package-lock.json` | 保留锁定依赖和 `test/build/preview` 脚本 |

模板已经实现以下交互：

- 方法、输入和输出折叠；
- 复杂字段 Hover 提示；
- 所有已展开来源方法的调用线；
- 展开方法通过“查看入线 / 查看出线”聚焦当前已绘制线路；
- 确定、条件和异常路径样式；
- 点击调用目标后展开并居中；
- 点击当前已绘制线路的箭头后反向追溯来源方法；
- 连续跳转的 Esc 返回栈；
- 普通滚轮纵向移动、`Alt + 滚轮`缩放、拖动和复位；
- 布局变化后的连线重绘和运行错误显示。

SOP 规定模板的使用流程、数据契约和验收标准；`map-data.js` 提供准确的数据写法；`main.js`、`call-lines.js` 和 `styles.css` 是内部实现依据。只有需要改变对应行为或排查问题时，才阅读内部实现。

模板当前按“一个根目录和三个直接子目录”展示。真实目录更少或更多时调整复制件的网格；存在更深目录时，只按本次真实层级扩展复制件的目录渲染和样式，再用 Playwright 重新验收。

模板已经统一采用自动正交路由，简单地图和复杂地图共用同一实现，不再让每个复制件重新选择或实现布线算法。仍需识别以下复杂路由特征，用于决定浏览器验收需要覆盖的展开规模：

- 一个来源方法存在多个调用步骤，展开后形成明显高扇出；
- 同一目标方法被调用两次以上；
- 同一个步骤包含两个以上调用；
- 同一目录内存在纵向堆叠文件，线路需要绕过同列正文；
- 同时展开两个来源方法后，出现共享端点、重合线段、穿过正文或走廊容量不足。

自动正交路由为每条调用分配独立来源行锚点，为同一目标分配多个端口，按线路纵向区间冲突分配物理通道，并按实际通道数动态扩大目录间距或外侧留白。人工 `lane` 只作为稳定排序提示，不能作为避免重叠的机制。

## 3. 开始条件

开始前确认：

| 信息 | 说明 |
|---|---|
| 功能名称 | 用于页面标题和输出目录 |
| 仓库范围 | 需要分析的仓库和模块 |
| 唯一功能入口 | 用户指定时直接采用；未指定时根据功能触发方式定位，无法唯一确定时再确认 |

同时记录业务源码的当前差异，完成后用来判断本任务是否误改业务代码。

### 建立实施问题记录

开始实施时新建一份过程文档：

```text
docs/<功能名>-code-map-retrospective.md
```

实施过程中遇到会影响代码分析、模板适用性、布局、调用线、构建或浏览器验收的问题时，在解决并验证后立即追加记录，不等到任务结束再凭记忆整理。统一格式：

```markdown
### 问题标题

- 问题现象：实际看到了什么，会造成什么影响。
- 原因：根据源码、页面或工具输出确认的原因。
- 解决方案：最终采用了什么修改或处理方式。
- 验证结果：通过什么检查确认问题已经解决。
- 可复用经验：下次生成其他代码地图时可以直接采用的结论。
```

只记录真实发生且有复用价值的问题，不记录正常步骤、普通进度或未经验证的猜测。尚未解决的问题标记为“未解决”，说明已有证据和下一步最小验证动作；本次没有实质问题时不添加占位问题或结论。最终方案替换了中间方案时，必须同步改写原记录中的解决方案、验证结果和可复用经验，不能让复盘停留在已经废弃的实现上。

## 4. 标准实施流程

### 步骤一：从单一入口递归分析

只选择一个功能入口，并从该入口向下递归追踪实际执行路径。

对每个尚未分析的方法：

1. 读取完整方法和必要上下文；
2. 记录它调用的全部项目自有方法；
3. 把尚未访问的目标加入待分析清单；
4. 遇到已经访问的方法时不重复创建节点，只记录调用关系；
5. 继续处理条件、循环、异常、`finally`、实际触发的构造方法、上下文方法和回调。

第三方库、Python 或其他语言内置能力、操作系统、数据库引擎、远程服务和硬件驱动的内部实现停止展开，只在当前方法的步骤中解释其作用。

动态目标能从源码、类型或配置确认时正常纳入；无法确认唯一目标时不猜测、不画正式调用线，记录为“未确认调用”。

分析完成后记录：

```text
功能入口：
目录数量：
文件数量：
方法数量：
调用数量：
未确认调用：
单个方法最大出调用数：
同一目标最大重复调用数：
同一步骤最大调用数：
```

根据后三项统计和真实文件排列判断属于简单路由还是复杂路由，并记录判断依据。

完成标准：所有页面方法都能从唯一入口沿项目内部调用到达，不存在为补图而加入的上游调用者或无关方法；调用规模和布局风险已经量化，不在绘图后才凭肉眼猜测复杂度。

### 步骤二：复制模板并确认适用性

在仓库根目录执行：

```powershell
Copy-Item -Recurse visualizations\code-map-template visualizations\<功能名>-code-map
```

复制后按照“模板使用契约”修改新目录。先比较真实目录层级与模板边界，明确本次是否只需替换数据和样式，还是必须扩展复制件的目录渲染。

同时完成路由适用性判断：

| 场景 | 处理方式 |
|---|---|
| 简单路由 | 直接使用模板自动路由，仍执行基础几何检查 |
| 复杂路由 | 直接使用模板自动路由，并扩大多来源展开和重复目标端口的验收范围 |
| 目标目录与来源目录不相邻 | 先明确合法的跨目录走廊；没有安全走廊时停止并调整布局 |
| 目录或文件层级超出模板 | 扩展复制件的渲染与网格，再重新计算路由走廊 |

模板已经把纯计算逻辑与 DOM 测量、SVG 绘制分离。复制后先运行已有路由测试；只有本次确实修改了通道或端口分配算法时，才补充直接覆盖新增规则的最小测试。

#### 自动正交路由最小契约

复杂路由实现至少遵守以下顺序：

1. 渲染调用列表时，把唯一 `call.id` 写到对应调用行的 `data-call-id`，来源坐标取该调用行面向走廊一侧的中点；
2. 按“目标方法 + 目标侧”统计端口数量，必要时先增加目标标题最小高度，确保端口间距可实现；
3. 测量来源行、目标标题和目录边界，确定每条线路唯一可用的目录间走廊或外侧走廊；
4. 同一目标的端口按来源纵坐标稳定排序；单端口居中，多端口在目标边界内等距分布；
5. 把每条线路表示为来源纵坐标到目标端口纵坐标的闭区间，并在两端加入安全余量；
6. 在同一物理走廊内按区间起点稳定排序，使用编号最小的无冲突通道；相交或安全余量接触的区间不得复用通道；
7. 按 `两侧边距 + 通道数 × 通道间距` 计算走廊最小宽度，更新目录间距或根容器外侧留白后重新测量；
8. 使用 `来源 → 水平进入通道 → 垂直到目标高度 → 水平进入目标端口` 的正交路径绘制，每条 SVG 路径保留 `data-call-id`；
9. 来源关闭、无调用或绘制失败时，同时清除路径、目标临时高度和动态走廊变量，不能把上一次布局状态留给下一次展开。

以下情况必须明确报错，不能静默兜底：调用行或目标不存在、重复调用 ID、坐标不是有限数、目标端口空间不足、`targetSide` 没有面向合法走廊、跨越非相邻目录且没有已定义走廊。

完成标准：新目录保留模板全部文件，不依赖 Panorama 页面或其他地图运行。

### 步骤三：填写真实数据和中文说明

`map-data.js` 使用扁平数组和 ID 建立关系。模板中的示例对象是填写格式参考。

| 对象 | 必需字段 |
|---|---|
| 目录 | `id, name, description, parentId` |
| 文件 | `id, directoryId, name, path, description` |
| 方法 | `id, fileId, signature, meaning, inputs, output, source, defaultExpanded, steps` |
| 步骤 | `id, text, tone` |
| 调用 | `id, sourceMethodId, sourceStepId, targetMethodId, kind, lane, targetSide, label` |

数据填写规则：

- ID 在当前地图内保持唯一、稳定、可读；
- `signature` 保存真实完整签名，页面只截取并显示短方法名；
- 只有功能入口设置 `defaultExpanded: true`；
- `inputs` 逐参数记录原名和中文含义；无显式参数时用 `inputNote` 说明实例状态或外部输入；
- `output.summary` 解释返回值用途；结构化返回值继续拆解 `output.fields`；
- 复杂字段增加 `detail`，用于问号 hover 说明来源、单位、特殊值或生命周期；
- `source` 使用真实相对路径和方法起止行号；
- `steps` 按真实执行顺序编号，总结条件、循环、异常和资源释放，不逐行翻译源码；
- 调用必须挂到真实发生调用的 `sourceStepId` 上。
- `targetSide` 表示目标端口面向的物理走廊，必须与来源和目标目录的位置关系一致；不能只为视觉好看任意填写；
- `lane` 只作为自动路由的确定性排序提示，不直接代表最终通道坐标，也不能用于人工绕过目录；

步骤样式 `tone`：

```text
normal      普通执行
condition   条件或分支
loop        循环
finally     必定执行的清理
error       异常路径
```

调用样式：

- `kind: certain`：执行到该步骤时确定调用，显示实线；
- `kind: conditional`：满足真实条件时调用，显示虚线；
- 红色由来源步骤的 `tone: error` 决定，不设置 `kind: error`；
- 无法确认的调用不写入 `calls`。

方法卡片展开后必须包含默认收起的输入和输出、编号逻辑、调用按钮以及源码位置。中文说明应让没有阅读源码的新手理解数据从哪里来、如何处理、调用了谁和最终得到什么。

完成标准：目录、文件、方法、字段和调用均来自真实源码，且静态数据能完整表达入口的项目内部调用闭包。

### 步骤四：进行必要的静态一致性检查

使用 AST 或目标语言的等价方式检查：

1. 每个方法确实存在于记录的文件中；
2. 参数名称、顺序和源码起止行号正确；
3. 只有一个方法默认展开，且它就是功能入口；
4. 所有目录、文件、方法、步骤和调用 ID 唯一；
5. 每条调用的来源方法、来源步骤和目标方法都存在；
6. 页面中的项目自有方法均能从入口到达。

复杂路由地图还要静态检查：

7. 每条调用都有唯一 `call.id`，并能绑定到页面中的一条真实调用列表行；
8. `targetSide` 面向可用走廊，跨目录调用没有错误地指向目录外侧；
9. 区间通道分配对输入顺序保持确定性，不修改输入数据；
10. 相交或仅在安全余量内分离的纵向区间不能复用同一通道；
11. 目标端口按来源位置稳定排序，满足最小间距并保持在目标标题边界内。

调用关系还必须回到实际调用点人工核对，因为 AST 中存在目标名称不等于运行时一定调用该目标。无法确认的关系继续保留在交付说明中。

这里不建立新的通用校验平台或测试框架；只对当前地图数据做一次可重复的静态核对。模板路由测试必须保留并执行，至少覆盖区间冲突、通道复用、确定性、重复 ID、非法坐标、目标端口间距和单目标居中；修改算法时再补充直接对应新规则的测试。

完成标准：静态数据没有路径、行号、参数、引用或入口状态错误。

### 步骤五：构建并使用 Playwright 验收

在新地图目录中执行：

```powershell
npm ci
npm run build
npm run preview
```

Windows 上执行 `npm ci` 前，先确认没有命令行指向当前地图目录的 Vite 开发或预览进程。若原生 `.node` 文件报 `EPERM`，应按进程命令行定位并只停止占用当前地图的进程，再重新安装；不要结束所有 Node 进程，也不要把反复删除 `node_modules` 当作解决方案。该问题真实发生时按“实施问题记录”格式写入复盘。

必须实际打开页面验收，不能只以构建成功作为完成标准。AI 执行时，如果环境提供 Playwright skill，应先完整阅读其 `SKILL.md` 并按其中流程操作；否则使用本地 Playwright CLI 或等价的真实 Chromium 自动化方式。

最低验收环境：Chromium、1440×900、浏览器缩放 100%。

Playwright 检查：

1. 页面标题、目录、文件、方法数量与分析结果一致；
2. 只有入口默认展开，所有输入和输出默认收起；
3. 方法标题与调用按钮只显示短方法名；
4. 输入输出可独立展开，复杂字段说明只在 hover 时出现并在移开后隐藏；
5. 同时展开多个来源方法时保留全部调用线，关闭一个来源时只删除它发出的线；
6. 展开方法的“查看入线 / 查看出线”只聚焦当前已绘制线路；再次点击取消，切换方向直接替换，收起聚焦方法自动清除；
7. 聚焦只改变颜色、粗细和透明度，不改变线路数量、路径坐标、实线/虚线样式或自动路由结果；取消后恢复正常与异常线路的原始颜色；
8. 每条当前已绘制线路都有且只有一个箭头点击区；点击箭头只反向跳转到已展开的来源方法，不改变任何方法的展开状态，按 Esc 返回箭头所在的目标方法；整条线路不可点击，此交互不受入线/出线聚焦影响；
9. 点击调用目标后目标展开并居中，连续跳转和连续 Esc 按后进先出返回；目标原本展开时返回不会误收起；箭头反向追溯与调用目标正向跳转混用时返回顺序仍正确；
10. 普通滚轮只纵向移动，`Alt + 滚轮`只缩放，拖动画布、缩放按钮和复位正常；
11. 确定调用为实线、条件调用为虚线、异常步骤发出的调用线为红色；
12. 节点不重叠，文字不溢出，容器包住子节点，调用线不穿过非来源、非目标的方法正文；
13. 对每种需要支持的展开状态，SVG 调用线数量等于所有已展开来源方法的调用总数，且 `data-call-id` 唯一；箭头点击区数量与调用线数量相等；
14. 每条调用的起点落在自己的调用列表行上，不同调用不共享完全相同的源锚点；
15. 重复目标使用不同目标端口，端口间距不小于箭头尺寸或设计安全间距；
16. 同一物理走廊中，纵向区间相交或安全余量接触的线路不复用通道；所有通道坐标位于走廊留白内；
17. 至少验证“只展开入口”和“入口加一个高调用量目标”两种状态；如果地图允许继续跳转，再验证最大合理展开状态；
18. 错误面板为空，页面自身的浏览器控制台没有错误或警告。

至少保存一张桌面视口截图进行视觉检查。几何断言用于证明端点、通道和正文边界没有冲突，截图用于判断线束密度、可读性和视觉节奏，两者不能互相替代。

发现布局问题时，可调整复制件的 CSS、目录顺序、列宽、画布宽度或 `targetSide`。如果目标目录与来源目录不相邻，应优先重排目录，让真实调用面向合法走廊；不能用人工 `lane` 绕过中间目录。只有确认模板的锚点、端口或通道算法本身违反契约时才修改算法，修改后重新运行路由测试、生产构建、几何检查和交互检查。

完成标准：生产构建通过，以上状态和交互均在真实浏览器中验证通过。

### 步骤六：最终复核和交付

1. 对最后一次页面调整重新执行生产构建；
2. 比较任务开始和结束时的业务源码差异；
3. 清理浏览器快照、截图、临时服务和不需要交付的构建临时文件；
4. 检查实施问题记录只包含实际问题、最终解决方案和验证结论，不保留已经被替换的中间结论；
5. 交付时说明：
   - 生成位置；
   - 安装、开发启动和生产构建方式；
   - 采用的唯一功能入口；
   - 覆盖的目录、文件、方法和调用数量；
   - 未确认调用，没有时明确写“无”；
   - 第三方和系统内部实现的停止边界；
   - 实施问题记录的位置及仍未解决的问题。

完成标准：接收者能够独立启动页面，并能从唯一入口沿图理解完整的项目内部执行流程。

- [ ] 已统计最大出调用数、重复目标数和同一步骤调用数，并完成简单/复杂路由判断。

- [ ] 实施问题记录已同步更新，只包含真实问题和经过验证的解决方案。
