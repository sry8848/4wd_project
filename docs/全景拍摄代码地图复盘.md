# 全景拍摄代码地图复盘

本文只记录实现过程中真实遇到的问题、解决方式和以后可以复用的经验。

## 实际问题与解决方式

### 1. 初始方案过度工程化

**问题：** 一次性的代码讲解网页被拆成了大量设计文档、契约校验、单元测试和多轮评审，实施成本超过了网页本身。

**解决方式：** 删除过程性设计文档、通用校验器和测试，只保留真实代码数据、网页实现、浏览器检查和本复盘文档。

### 2. 嵌套布局和调用线属于两种布局系统

**问题：** 目录、文件和方法适合由浏览器文档流自然排版；调用线需要跨容器连接具体步骤。如果把两者都交给图布局引擎，容器尺寸和折叠状态会变得难以控制。

**解决方式：** 卡片使用普通 HTML/CSS 布局，调用线单独使用覆盖在同一画布上的 SVG。方法展开后读取真实 DOM 位置，再绘制固定通道的正交折线。

### 3. 缩放后重绘容易出现坐标偏移

**问题：** Panzoom 会给整个画布添加变换，直接把 `getBoundingClientRect()` 的屏幕坐标写入 SVG 会导致连线错位。

**解决方式：** 使用 SVG 的 `getScreenCTM().inverse()` 把屏幕坐标转换回 SVG 用户坐标。平移和缩放时 SVG 与卡片一起变换，不重复重绘；只在内容尺寸变化时重绘。

### 4. 自动避障连线的收益不够高

**问题：** 为 28 条固定调用实现通用碰撞检测和自动路由，会引入大量难以稳定验证的算法代码。

**解决方式：** 为每条调用预先指定通道编号和目标侧，CSS 为通道预留空白。最终通过浏览器实际检查调整少量通道，而不是建设通用布局系统。

### 5. 静态分析数据容易把“可能调用”写成“确定调用”

**问题：** `_capture_frames()` 的 `_frame_path()` 只在非 OpenCV 分支执行，初次整理时曾被标记为确定调用。

**解决方式：** 回到真实分支结构核对，将其标记为条件调用；同时明确 `__init__()` 没有显式返回值，避免给新手错误的 Python 对象模型印象。

### 6. Windows 管道输出存在编码差异

**问题：** Python 用系统默认 GBK 解码 Node.js 输出的 UTF-8 中文 JSON 时触发 `UnicodeDecodeError`，导致第一次 AST 核对失败；这不是数据错误。

**解决方式：** 以字节方式接收子进程输出，再明确使用 UTF-8 解码。随后成功核对 4 个目录、4 个文件、24 个方法、28 条调用以及全部方法起止行号。

### 7. Playwright 包装脚本在 Windows/WSL 中受 CRLF 影响

**问题：** skill 提供的 shell 包装脚本在 WSL 中被识别为 `bash\r`，无法执行；直接修改系统级 skill 文件不属于本任务范围。

**解决方式：** 使用本机 npx 执行包装脚本内部相同的 `@playwright/cli` 命令，继续使用命名会话、快照和真实浏览器检查流程。

### 8. 默认缩放中心会把大画布推离视口

**问题：** Panzoom 默认以 HTML 元素中心作为 `transform-origin`。代码地图宽度远大于视口，连续点击“缩小”后，内容向右下方移动并留下大面积空白。

**解决方式：** 将画布变换原点固定为左上角 `0 0`，再根据当前缩放、平移和鼠标位置计算缩放后的新平移量。普通滚轮只调整纵向平移，`Alt + 滚轮`和缩放按钮共用同一套焦点缩放逻辑。

### 9. 浏览器会自动请求 favicon

**问题：** 首次 Playwright 检查只有一条控制台错误：浏览器请求缺失的 `favicon.ico` 得到 404，与页面功能无关但会污染诊断结果。

**解决方式：** 在 HTML 中声明内嵌空图标，避免产生额外网络请求；重新加载后的控制台为 0 错误、0 警告。

### 10. 参数说明既要详细，也不能默认铺满卡片

**问题：** 把复杂概念全部写在参数行中会让 `capture_panorama()` 的 21 个参数难以扫描；只写“会话目录、候选帧”等短语，新手又无法理解。

**解决方式：** 每个参数先显示原名和一句直接解释，只给复杂概念增加问号提示。后续根据实际阅读需求进一步精简为仅鼠标悬停显示，点击不保存状态，简单字段不增加额外层级。

### 11. 提示框朝下展开会被画布边缘截断

**问题：** 首个参数位于视口底部时，向下弹出的详细说明只有边框可见，正文落在裁剪区域外。

**解决方式：** 提示框改为从问号上方向上展开，并提高当前方法卡片的层级；复查后正文完整可见，也不会改变卡片尺寸和调用线位置。

### 12. 单一“当前方法”状态不符合多方法对照阅读

**问题：** 原实现打开新方法就替换 `activeMethodId`，导致先前展开方法的调用线消失。

**解决方式：** 每次重绘直接读取所有带 `open` 的方法，汇总它们发出的调用；关闭一个方法时，自然只移除该来源的调用线。调用目标跳转同样只负责展开和居中，不关闭来源方法。

### 13. 输入输出默认展开会挤占主要逻辑

**问题：** 参数和返回字段解释补全后内容明显变长，方法一展开就展示全部输入输出，会把真正需要顺序阅读的执行逻辑推到很远的位置。

**解决方式：** 输入和输出分别改成独立的原生 `<details>`，默认收起且标题不显示数量；任何展开或收起都会在下一帧重新计算调用线。方法标题已经足以定位代码，因此同时删除重复的“完整签名”。

### 14. 只展示公开流程会隐藏关键项目逻辑

**问题：** 原来的 24 个方法覆盖了业务主流程，却遗漏了角度转占空比、候选帧清晰度选择、摄像头属性设置和后端分派等项目自己实现的逻辑，新手仍会在关键细节处遇到断点。

**解决方式：** 沿真实调用继续追踪并补充 14 个可达方法和 19 条调用关系。第三方库、系统命令和硬件驱动只在步骤中解释，不伪造项目方法节点。其中 `ServoController._angle_to_duty()` 明确展示源码中的线性公式 `min_duty + (max_duty - min_duty) × angle ÷ 180`。

### 15. `Ctrl + Tab` 不能作为网页内返回快捷键

**问题：** Chrome 和 Edge 会优先用 `Ctrl + Tab` 切换浏览器标签页，页面脚本无法稳定截获，因此不能满足可靠返回的要求。

**解决方式：** 改用 Esc，并为每次调用跳转记录来源、目标及目标原本是否展开。连续跳转按后进先出返回；只有本次跳转打开的目标才会被收起，原本已经展开的目标会保留。

## 最终检查结果

- Vite 生产构建通过。
- Playwright 在桌面视口完成字段提示、连续方法跳转、Esc 返回栈、多方法连线、普通滚轮、`Alt + 滚轮`、按钮缩放和复位检查。
- 页面渲染 4 个目录、4 个文件、38 个方法和 47 条调用关系；方法标题和调用按钮都只显示简短方法名。
- `capture_panorama()` 展示 21 个真实参数和 `PanoramaResult` 的 4 个字段，参数名已与 Python AST 核对。
- 初始状态只有 `capture_panorama()` 展开，输入和输出均收起，页面不再出现“完整签名”。
- 连续跳转 `capture_panorama()` → `ServoController.move_to()` → `ServoController._angle_to_duty()` 后，两次 Esc 会依次返回；跳转前已展开的目标不会被误收起。
- 方法卡片、节点标题、字段和逻辑步骤没有内部内容溢出或相互重叠，控制台无错误或警告。

## 可复用经验

- 一次性教学可视化优先采用“静态真实数据 + DOM 布局 + 少量 SVG”，不要先建设通用平台。
- 动态连线最重要的是统一坐标系，其次才是路径样式。
- 业务入口实际可达且由项目自己实现的关键调用应进入图中；第三方内部实现留在文字解释里，避免伪造调用节点。
- 条件分支、异常和 `finally` 必须根据源码语义区分，不能只看是否出现了函数名。
- 最终质量应以真实浏览器中的文字、容器和连线为准；过程文档和测试数量不是交付目标。
- Windows 上跨进程传递中文文本时不要依赖系统默认编码，应明确约定 UTF-8。
