# 摄像头云台视觉任务重构详细执行计划

## 1. 文档状态

本文是待执行的重构与实机验证计划，不代表其中功能已经实现或通过实机验证。

计划制定时间：2026-07-14。

当前代码基线：

- 已有 `src/tasks/camera_servo_scan.py`，按上下、左右角度网格执行公共扫描。
- 二维码已通过 `src/tasks/qr_servo_scan.py` 使用公共扫描器。
- 颜色、人脸和逐方向拍照的云台业务流程仍主要位于 `src/tools/`。
- 摄像头左右舵机为 BCM 11 / J2，上下舵机为 BCM 9 / J3。
- 当前命令行将两个轴统一限制在 20～160 度，但这不是两个轴分别完成机械标定后的结论。
- 当前障碍照片在小车倒车恢复到起点节点后拍摄，不是发现障碍并刹车时的现场画面。
- 当前 Windows 测试环境缺少 `pytest` 和 `fastapi`，完整测试发现会出现导入错误；不得把当前状态描述为“全部测试通过”。

## 2. 改造目标

本次改造的目标不是让人脸、二维码、颜色、拍照和障碍记录使用完全相同的搜索策略，而是建立一套稳定的摄像头云台基础能力，让不同任务复用硬件控制和扫描执行过程，同时保留各自不同的业务逻辑。

最终应形成以下结构：

```text
CameraGimbal
    -> 只负责摄像头双轴云台硬件

CameraServoScanner
    -> 只负责按有序位置移动、取帧、超时和取消

QRCodeServoScanner
    -> 定义二维码解码及 TYPE:ID 成功条件

ColorServoScanner
    -> 定义颜色稳定确认条件

FaceServoSearch
    -> 定义人脸搜索、居中和身份确认状态

CameraPhotoScan
    -> 定义遍历全部位置并保存照片

ObstacleEvidenceRecorder
    -> 使用超声波事件触发现场取帧，不执行视觉盲扫
```

## 3. 本质需求

### 3.1 真正可以共享的能力

以下流程与识别目标无关，可以共享：

1. 校验云台位置是否在安全范围内。
2. 将摄像头转到指定左右和上下角度。
3. 等待舵机稳定。
4. 丢弃舵机运动期间产生的模糊帧或旧缓存帧。
5. 读取稳定后的新画面。
6. 处理任务超时、取消和运行统计。
7. 找到目标后停止执行后续位置。
8. 异常退出时释放摄像头和云台拥有的资源。

### 3.2 不能统一的业务逻辑

以下逻辑必须留在具体任务中：

- 人脸：发现人脸、画面居中、身份确认、目标丢失恢复。
- 二维码：解码、`TYPE:ID` 格式校验、无效内容记录。
- 颜色：目标面积判断、主颜色稳定确认。
- 逐方向拍照：必须遍历全部位置，没有“找到即停止”。
- 障碍记录：超声波已经提供触发事件，不应再执行完整视觉搜索。

## 4. 范围与非目标

### 4.1 本次范围

- 标定摄像头云台两个轴的安全范围和中心位置。
- 建立完整的 `CameraGimbal` 硬件边界。
- 将公共扫描器改为按明确的有序位置执行。
- 将颜色、人脸和逐方向拍照流程从工具层下沉到任务层。
- 保留二维码专属任务并迁移到新扫描接口。
- 为人脸增加“搜索—居中—确认—完成”的有限状态流程。
- 将障碍现场取帧提前到发现障碍并刹车之后、倒车恢复之前。
- 更新本地自动检查、树莓派实机步骤和项目文档。

### 4.2 本次不做

- 不更换 Haar、LBPH、OpenCV 二维码和 HSV 颜色识别算法。
- 不加入云端人脸识别、目标检测模型或新的 AI 服务。
- 不修改超声波阈值、连续判障规则、A* 或巡线算法。
- 不支持多人脸跟踪；第一版人脸搜索场景要求画面中只有一个主要人脸。
- 不增加持续无限跟踪模式；第一版在完成身份确认后结束任务。
- 不把摄像头、云台或存储逻辑写入电机、巡线、超声波硬件类。
- 不在树莓派使用 Git，只复制本地最新 `src/`。

## 5. 信任边界和资源契约

### 5.1 外部输入

命令行角度、GPIO、超时、帧数等属于系统外部输入，必须在初始化摄像头和 GPIO 前完成校验。

非法输入必须明确报错，不允许：

- 自动放大 `frames_per_position` 掩盖矛盾配置。
- 把越界角度静默截断到安全范围。
- 为同一角度提供多个字段名或多个来源兜底。

### 5.2 硬件资源所有权

- `CameraGimbal` 拥有左右和上下两个舵机及对应 GPIO。
- `CameraGimbal.close()` 只释放自己拥有的两个 GPIO，不调用全局 `GPIO.cleanup()`。
- `CameraServoScanner`、人脸、二维码、颜色和拍照任务不关闭硬件。
- `src/tools/` 或服务器入口层创建并最终释放摄像头和云台。
- 正常完成时可以由任务明确要求云台回中心；异常和 `Ctrl+C` 时优先停止 PWM 并释放资源，不在未知机械状态下强制执行新的回中动作。

### 5.3 导航安全契约

- 电机刹车优先于拍照。
- 摄像头失败不能阻止倒车恢复。
- 障碍证据取帧不能改变超声波封边结果。
- 障碍照片写盘不能插入正在执行的倒车动作。
- 导航运行期间摄像头云台必须保持正前方，不能与其他视觉任务并发争用。


### 8.2 配置变更

只有阶段 0 已确认的数据才写入 `src/config.py`：

```python
CAMERA_PAN_MIN_ANGLE = 实测值
CAMERA_PAN_CENTER_ANGLE = 实测值
CAMERA_PAN_MAX_ANGLE = 实测值

CAMERA_TILT_MIN_ANGLE = 实测值
CAMERA_TILT_CENTER_ANGLE = 实测值
CAMERA_TILT_MAX_ANGLE = 实测值
```

每个配置注释必须说明来自哪次实机验证，不继续沿用没有证据的统一范围。

### 8.3 数据契约

在 `camera_gimbal.py` 中定义不可变位置对象：

```python
@dataclass(frozen=True)
class GimbalPose:
    pan_angle: float
    tilt_angle: float
```

`CameraGimbal` 对外只提供必要能力：

```python
class CameraGimbal:
    def move_to(self, pose: GimbalPose): ...
    def center(self): ...
    def close(self): ...
```

不增加只转发一行调用的包装方法，例如 `move_left()`、`move_right()`、`look_up()`。

### 8.4 `move_to()` 分步逻辑

1. 同时校验左右、上下角度，任何一个越界时两个轴都不动作。
2. 与当前已知位置比较，只移动发生变化的轴。
3. 按固定且有文档说明的轴顺序移动，第一版保持现有“上下后左右”的行为，除非实机证明需要改变。
4. 两个轴完成后才更新当前位置。
5. 返回实际完成位置，供任务记录。

首次创建后当前位置属于未知状态，第一次 `move_to()` 必须明确设置两个轴。

### 8.5 `center()` 和 `close()` 契约

- `center()` 是显式业务动作，移动到阶段 0 确认的中心位置。
- `close()` 停止两个舵机 PWM，并分别释放 BCM 11、9。
- `close()` 不自动调用 `center()`，避免异常退出时产生新的机械动作。
- 即使关闭第二个舵机发生异常，也要尝试关闭另一个舵机。
- 禁止无参数 `GPIO.cleanup()`。

### 8.6 工具层变化

将 `enter_camera_servos()` 替换为单一的：

```python
enter_camera_gimbal(stack, args)
```

调用者只取得一个 `CameraGimbal`，不再分别持有两个舵机。

外部参数必须在调用该函数前校验，避免先初始化 GPIO 再发现参数错误。

### 8.7 自动检查

`tests/test_camera_gimbal.py` 至少覆盖：

1. 第一次移动会设置两个轴。
2. 只改变一个角度时只移动对应轴。
3. 任一角度越界时两个轴都不动作。
4. `center()` 使用配置的两个中心角度。
5. 两个轴使用相同 GPIO 编号时初始化失败。
6. `close()` 关闭并清理两个自有引脚。
7. 一个舵机关闭失败时仍尝试关闭另一个舵机。

这些测试使用假 GPIO 或假舵机对象，不增加测试专用生产字段和测试分支。

### 8.8 实机验证

- 中心、左右、上下位置与阶段 0 结论一致。
- 重复发送同一个位置时云台不重复抖动。
- 正常结束时按入口要求回中心。
- `Ctrl+C` 时不强制回中，只停止 PWM 并释放资源。
- 退出后可立即重新运行拍照工具。

## 9. 阶段 2：重构有序位置扫描执行器

### 9.1 目标文件

修改：

```text
src/tasks/camera_servo_scan.py
tests/test_camera_servo_scan.py
```

后续同一阶段必须同步修改全部调用者，不能保留旧、新两套扫描接口。

### 9.2 接口变化

删除扫描器直接接收 `pan_angles`、`tilt_angles` 并生成矩形组合的职责。

新接口接收明确的有序位置：

```python
result = scanner.scan(
    frame_handler,
    poses=(
        GimbalPose(90, 90),
        GimbalPose(70, 90),
        GimbalPose(70, 75),
    ),
    frames_per_pose=10,
    discard_frames_after_move=3,
    timeout_seconds=30,
    cancel_requested_fn=cancel_requested,
)
```

命令行仍可保留便于新手输入的 `--pan-angles` 和 `--tilt-angles`，但具体任务负责将它们转换为适合本任务的有序位置。公共执行器不负责决定搜索策略。

### 9.3 执行顺序

1. 在任何硬件动作前校验位置列表、帧数、丢弃帧数和超时。
2. 检查是否已经取消。
3. 移动云台到当前位置。
4. 检查超时和取消。
5. 丢弃运动期间的旧帧。
6. 逐帧调用 `frame_handler`。
7. `frame_handler` 返回非 `None` 时立即生成成功结果。
8. 成功后不再访问后续位置。
9. 位置用完但没有结果时返回未找到。
10. 超时或取消时返回明确状态，不把它们混成普通未找到。

### 9.4 结果契约

`CameraServoScanResult` 至少包含：

- `value`：具体任务返回的结果。
- `pose`：成功时或最后执行的位置。
- `positions_scanned`。
- `frames_scanned`。
- `elapsed_seconds`。
- `status`：`found`、`exhausted`、`timed_out` 或 `canceled`。
- `last_frame`：最后送入任务的有效画面。

结果值使用泛型类型，避免长期使用无约束的 `object`。不为测试添加额外字段。

### 9.5 停止和回中规则

- 扫描器停止的唯一业务信号是处理器返回结果。
- 扫描器停止后不自动回中心。
- 具体任务必须先保存成功画面或使用结果，再决定是否回中心。
- 正常结束时 standalone 工具默认回中心。
- 异常或 `Ctrl+C` 时不执行新的回中动作，只关闭资源。

### 9.6 自动检查

更新 `tests/test_camera_servo_scan.py`，至少覆盖：

1. 严格按照传入位置顺序移动。
2. 成功后不移动到下一个位置。
3. 每个位置只处理规定数量的帧。
4. 运动后的旧帧不进入任务处理器。
5. 超时发生后不继续移动。
6. 取消发生后不继续移动。
7. 空位置列表立即报错且不访问硬件。
8. 越界位置在第一步统一报错，不发生部分移动。
9. 正确区分 `exhausted`、`timed_out` 和 `canceled`。

### 9.7 阶段成功标准

- 公共执行器中不存在二维码、人脸、颜色、照片路径等业务概念。
- 所有调用者使用一个 `CameraGimbal`。
- 旧的 `pan_angles × tilt_angles` 内部循环被删除，不保留兼容分支。
- 二维码原有成功、失败诊断行为通过回归检查。

### 10.2 二维码任务

保留以下专属逻辑：

- 调用 `QRCodeRecognizer.decode(frame)`。
- 调用 `parse_qr_payload()` 校验 `TYPE:ID`。

- 第一次得到有效内容后返回结果。
- 超时或位置耗尽时保留最后一帧供诊断。

二维码任务不负责创建或关闭摄像头、云台。

### 10.3 颜色任务

`ColorServoScanner` 负责：

1. 在一个位置内持续调用 `ColorDetector.detect()`。
2. 只对满足面积阈值的主颜色计数。
3. 切换位置时清空颜色确认状态。
4. 达到稳定条件后返回 `ColorDetectionResult` 和对应位置。

参数契约：

```text
stable_frames >= 1
stable_frames <= frames_per_pose
```

非法组合必须在创建摄像头和云台前报错，不自动调整。

### 10.4 人脸任务的迁移阶段

本阶段只把当前固定角度确认逻辑移出工具层，暂不增加居中动作：

1. 在一个位置内调用 `LocalFaceRecognizer.recognize()`。
2. 选择当前画面中距离最小的已登记人员。
3. 切换位置时清空候选人和确认计数。
4. 达到身份确认条件后返回姓名、距离和位置。

参数契约：

```text
confirm_frames >= 1
confirm_frames <= frames_per_pose
```

第一版实机场景要求只出现一个主要人脸。多人脸目标选择不在本阶段解决。

### 10.5 逐方向拍照任务

`CameraPhotoScan` 与识别任务不同：

- 它必须遍历全部位置。
- 每个位置只保存最终稳定帧。
- 文件名包含扫描序号、左右角度和上下角度。
- 任一位置保存失败时返回明确失败，不生成虚假的完整成功统计。
- 全部完成后由入口显式调用 `gimbal.center()`。

### 10.6 工具入口最终职责

四个工具只保留：

1. 解析命令行。
2. 校验外部输入。
3. 创建识别算法对象。
4. 在一个 `ExitStack` 中创建摄像头和 `CameraGimbal`。
5. 调用对应任务。
6. 保存入口专属诊断结果并打印。
7. 正常完成时回中心。
8. 最外层统一释放资源。

工具中删除大段可变字典状态和逐帧业务判断，不能让工具继续承担任务层职责。

### 10.7 自动检查

二维码：

- 无内容时继续。
- 无效格式时记录但继续。
- 有效内容时停止。
- 成功画面与返回画面一致。

颜色：

- 同一位置达到稳定帧数后成功。
- 不同颜色会重置确认。
- 切换位置会重置确认。
- 稳定帧数大于位置帧预算时提前报错。

人脸：

- 同一人员达到确认帧数后成功。
- 人员标签切换时重新计数。
- 未知人员不能成功。
- 切换位置不能累计旧位置计数。
- 确认帧数大于位置帧预算时提前报错。

拍照：

- 每个位置只保存一张。
- 输出文件名唯一。
- 保存张数与位置数一致。
- 中途失败不能报告完整成功。

### 10.8 实机回归顺序

1. 逐方向拍照，验证机械位置和照片对应关系。
2. 二维码中心位置成功。
3. 二维码左、右位置成功。
4. 颜色中心、左、右位置成功。
5. 已登记人员中心、左、右位置成功。
6. 未登记人员不能被确认。
7. 每个工具执行结束后摄像头可以立即重新打开。

## 11. 阶段 4：实现人脸搜索、居中和身份确认

### 11.1 目标

人脸任务不能在第一帧出现人脸时就结束。完成条件必须同时满足：

1. 找到主要人脸。
2. 人脸位于画面中心容许区域。
3. 在居中后的稳定画面中确认同一登记人员。

### 11.2 状态定义

```text
SEARCH
    按粗略位置寻找主要人脸

CENTERING
    根据人脸框中心与画面中心的偏差小角度调整云台

CONFIRMING
    保持当前位置并确认身份

COMPLETE
    返回姓名、距离、最终位置和成功画面

EXHAUSTED / TIMED_OUT / CANCELED
    返回对应失败状态
```

### 11.3 SEARCH 行为

- 使用 `HaarFaceDetector` 查找主要人脸。
- 第一版以面积最大的脸作为主要目标。
- 如果画面出现多个面积接近的人脸，输出“多人脸场景不满足第一版契约”，不能静默选择一个身份。
- 找到主要人脸后不立即返回成功，进入 `CENTERING`。

### 11.4 CENTERING 行为

根据 `FaceBox` 计算人脸中心：

```text
face_center_x = x + width / 2
face_center_y = y + height / 2
```

与画面中心比较：

```text
error_x = face_center_x - frame_width / 2
error_y = face_center_y - frame_height / 2
```

居中参数第一版作为待实测任务参数，不直接写成硬件事实：

- 中心容许范围：画面宽高的约 10%～15%，具体值通过实机确定。
- 单次调整：先从 5 度开始实测。
- 每次移动后清空旧帧和身份确认窗口。
- 新角度必须经过 `CameraGimbal` 边界校验。
- 达到机械边界仍无法居中时返回明确失败，不在边界反复抖动。

云台角度与画面移动方向必须使用阶段 0 的实测方向映射，禁止凭经验假设正负号。

### 11.5 CONFIRMING 行为

居中后保持云台位置，使用短窗口确认身份。

建议第一版验证“最近 5 帧中至少 3 帧是同一登记人员”，原因是 Haar 检测可能偶尔漏掉一帧，严格连续帧容易被单帧抖动清零。

确认窗口规则：

- 只统计当前居中位置的画面。
- 云台再次移动时清空窗口。
- 未知人员记录为未知，不能借用上一帧标签。
- 两个不同登记人员不能合并计数。
- 达到 3/5 后进入 `COMPLETE`。

如果实机反例证明 3/5 容易误识别，再调整窗口；不能没有证据地不断堆叠阈值。

### 11.6 目标丢失

- `CENTERING` 中短暂丢失时，在当前角度读取少量新帧。
- 连续丢失达到限定帧数后返回最近搜索位置并继续后续搜索计划。
- 不建立复杂预测器，不根据历史速度猜测人脸位置。
- 超过总任务时限立即停止。

### 11.7 找到后如何停止

1. `FaceServoSearch` 返回成功结果。
2. 公共扫描执行器不再访问后续位置。
3. 入口在摄像头和云台仍打开时保存成功诊断画面。
4. 正常完成后显式调用 `gimbal.center()`。
5. 退出 `ExitStack`，停止 PWM 并释放资源。

成功前不得关闭云台，否则摄像头可能在保存画面前失去目标方向。

### 11.8 自动检查

- 人脸在左侧时产生正确方向的调整。
- 人脸在右侧时调整方向相反。
- 人脸在上、下位置时调整对应轴。
- 进入中心容许区后不继续抖动。
- 移动后清空身份确认窗口。
- 3/5 同一人员成功。
- 未知人员和两个不同人员不能成功。
- 目标持续丢失后恢复搜索。
- 到达角度边界后明确失败。
- 成功后不再执行任何搜索位置。

### 11.9 实机验收

- 一名已登记人员分别站在画面中心、偏左、偏右，均能被找到并居中。
- 未登记人员不能被输出为登记人员。
- 人员短暂转头不会立即错误成功或无限卡住。
- 云台没有在中心附近持续左右抖动。
- 成功画面中人脸位于设定中心容许范围。

### 11.10 建议提交

```text

```

## 12. 阶段 5：调整障碍现场取帧和持久化时序

### 12.1 当前问题

当前 `GridNavigator` 在确认边被阻塞后先执行倒车恢复，`obstacle_result_fn` 在恢复结束后才调用 `ObstacleRecorder.record()`，因此保存的画面是恢复后的现场。

如果障碍是行人或其他动态物体，倒车期间它可能已经离开，最终照片可能只剩空路，不能准确说明触发超声波判障时看到了什么。

### 12.2 目标时序

```text
边执行返回 blocked_on_planned_edge
    ↓
立即确认电机已经 brake
    ↓
调用 obstacle_detected_fn
    ↓
摄像头从正前方读取并复制一帧到内存
    ↓
执行 recover_to_start_node
    ↓
恢复完成或失败，电机停止
    ↓
调用 obstacle_result_fn
    ↓
将此前缓存的画面和最终恢复状态写入磁盘
```

### 12.3 目标文件

预计修改：

```text
src/tasks/grid_navigation.py
src/server/ride_service.py
src/server/camera_runtime.py
src/server/obstacle_recorder.py
src/server/obstacle_store.py
tests/test_grid_navigation.py 或对应现有导航测试
tests/test_ride_service.py
tests/test_camera_runtime.py
tests/test_obstacle_recorder.py
```

修改前必须重新核对实际测试文件名，不能创建与现有测试重复的平行文件。

### 12.4 回调拆分

在 `GridNavigator.navigate()` 中增加发现阶段回调：

```text
obstacle_detected_fn(
    from_node,
    to_node,
    distance_cm,
    heading,
)
```

保留恢复结束回调：

```text
obstacle_result_fn(
    from_node,
    to_node,
    distance_cm,
    recovery_status,
    final_heading,
)
```

发现阶段回调必须在 `recover_to_start_node()` 之前执行，并且测试必须证明电机刹车发生在回调之前。

### 12.5 待保存证据对象

定义不可变的 `PendingObstacleEvidence`，至少包含：

- 记录 ID。
- 发现时间。
- 行程 ID。
- 起点和目标点。
- 确认距离。
- 缓存画面或空值。
- 取帧错误或空值。

第一版导航同一时刻只处理一条边，`RideService` 在本次导航调用范围内只保存一个待完成证据对象，不建立全局队列或数据库。

### 12.6 摄像头运行时能力

`BackendCamera` 增加明确的“读取并复制当前帧”能力，不能复用写盘方法伪装成内存取帧。

要求：

- 丢弃至少一个可能的旧缓存帧。
- 返回独立画面副本，后续摄像头读取不能覆盖它。
- 读取失败后记录错误，但不能向导航传播为运动失败。
- 不在 `GridNavigator` 中 import `cv2`。

### 12.7 障碍记录器拆分

`ObstacleRecorder` 调整为两个明确阶段：

```python
pending = recorder.capture_evidence(...)
record = recorder.save_result(pending, recovery_status=...)
```

`capture_evidence()`：

- 在发现障碍时分配记录 ID 和发现时间。
- 尝试缓存画面。
- 不写 JSON/JPEG。
- 不参与导航决策。

`save_result()`：

- 在恢复结束、运动停止后执行。
- 保存此前缓存的画面。
- 保存恢复状态、恢复节点和取帧错误。
- 即使恢复失败，只要检测时已获得画面，也允许保存真实现场照片。
- 没有画面时不生成占位 JPEG，只保存明确错误。

### 12.8 云台方向契约

导航开始前：

- 摄像头云台必须回到阶段 0 标定的中心位置。
- 运行中的叫车任务独占摄像头和云台。
- 障碍发生时不执行 15 个方向扫描，也不临时大范围转动。

若未来超声波改为可转向扫描，再单独建立超声波方向与摄像头姿态的实机标定。本阶段不假设两个舵机角度可以直接互换。

### 12.9 错误优先级

```text
最高：电机刹车和恢复安全
其次：保存导航状态和障碍边
最后：摄像头证据
```

摄像头读取或写盘失败必须满足：

- 不改变 `dynamic_blocked_edges`。
- 不阻止 `recover_to_start_node()`。
- 不把导航成功误报为失败。
- 在障碍记录或系统消息中保存真实错误原因。

### 12.10 自动检查

1. 阻塞结果后先刹车，再调用发现回调。
2. 发现回调发生在倒车恢复之前。
3. 摄像头异常时仍调用恢复。
4. 恢复完成后才执行写盘。
5. 恢复失败时仍能保存此前缓存的真实画面。
6. 没有缓存画面时不生成 JPEG。
7. 记录时间使用发现障碍的时间。
8. 原有封边、重新规划和行程消息行为保持不变。
9. 取消请求仍优先停止导航，不因证据流程延迟。

### 12.11 实机验收

- 使用静态纸箱触发障碍，照片内容来自倒车前现场。
- 使用人员短暂停留触发，验证动态障碍离开后仍保留发现时画面。
- 人为占用或断开摄像头，验证小车仍刹车、倒车恢复并记录错误。
- 恢复成功和恢复失败两种记录均可在前端查询。
- 后端重启后已保存记录仍可读取。

### 12.12 建议提交

```text

```

## 15. 风险与应对

### 15.1 机械范围判断错误

影响：撞限位、舵机抖动、线材拉扯。

应对：阶段 0 先实测；每次只扩大 10 度；配置两个轴独立范围。

### 15.2 摄像头与云台资源竞争

影响：摄像头打不开或导航期间云台被其他任务转走。

应对：同一进程单一资源所有者；实机工具运行前停止后端；导航期间禁止并发视觉任务。

### 15.3 重构与行为升级混在一起

影响：失败时无法判断是接口迁移还是新算法问题。

应对：阶段 3 先保持现有识别语义，阶段 4 再增加人脸居中和窗口确认。

### 15.4 障碍证据流程拖慢恢复

影响：小车在障碍前停留过久。

应对：刹车后只复制内存帧，不写盘；恢复停止后才持久化。

### 15.5 摄像头失败扩散到导航

影响：因为拍照失败而没有执行安全恢复。

应对：发现回调内部捕获摄像头错误并返回带错误的待保存证据；导航不依赖取帧成功。

### 15.6 人脸居中来回抖动

影响：无法进入身份确认状态。

应对：中心容许区、小角度调整、每次移动后等待稳定；参数以实机现象调整，不增加预测和复杂控制器。

## 16. 执行过程中的决策闸门

以下情况必须暂停并向用户报告，不能自行继续：

1. 两个轴的安全范围或方向无法通过实机确定。
2. 多人脸是否需要支持，且该需求会改变目标选择规则。
3. 导航期间还需要并发运行人脸或二维码功能。
4. 障碍恢复失败时是否允许前端展示现场照片，若现有产品要求与本计划冲突。
5. 完整测试需要安装新依赖。
6. 当前无关未提交文档与本任务修改发生内容冲突。
7. 需要修改前端接口字段或产生破坏性 API 变更。

## 17. 最终完成定义

只有以下条件全部满足，才能认为本计划执行完成：

- `CameraGimbal` 成为摄像头双舵机的唯一硬件边界。
- 公共扫描器只接收有序位置，不包含具体识别业务。
- 二维码、颜色、人脸和逐方向拍照分别位于任务层。
- 人脸完成搜索、居中和身份确认后停止。
- 障碍现场帧在刹车后、倒车前取得，恢复后持久化。
- 所有正常和异常路径都有明确资源释放行为。
- 相关自动检查真实执行并记录结果。
- 树莓派实机测试矩阵完成，没有未解释的失败。
- 文档与当前代码、实机结果一致。
- Git 提交符合 Conventional Commits，未混入无关改动。
