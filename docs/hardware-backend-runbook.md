# 前后端实车模式运行说明

## 1. 启动前条件

1. 先用手动工具确认电机、巡线、超声波和相邻节点导航正常。
2. 把小车准确放在一个可信网格节点中心，并确认车头朝向。
3. 实车模式只能启动一个 Uvicorn worker，禁止使用 `--reload`。
4. 保持终端可操作；异常时先按 `Ctrl+C`，应用关闭流程会取消行程、刹车并释放 GPIO。

树莓派当前使用 Python 3.7，必须安装仓库锁定的兼容依赖，不能直接安装最新版：

```bash
python3 -m pip install --user -r requirements-rpi.txt
python3 -c "import fastapi, uvicorn; print(fastapi.__version__, uvicorn.__version__)"
```

预期版本为 FastAPI `0.103.2`、Uvicorn `0.21.1`。

## 2. 启动实车模式

下面示例表示小车实际位于 `A1`，车头朝东：

```bash
export CAR_INITIAL_POSITION=A1
export CAR_INITIAL_HEADING=east
python3 -m uvicorn src.server.app:app --host 0.0.0.0 --port 8000 --workers 1
```

当前后端只支持实车运行，不再提供 `fake` 模式。缺少初始点位或朝向时，后端必须拒绝启动。

启动后先检查：

```bash
curl http://127.0.0.1:8000/api/health
```

成功标准：

```text
navigation_mode = hardware
hardware_ready = true
```

## 3. 首次前后端实车验收

### 3.1 安全边界

1. 先清空路线上的人员和杂物，保证小车前方有足够停车距离。
2. 一人观察小车，一人保持 Uvicorn 终端可操作；异常时立即按 `Ctrl+C`。
3. 首次联调不添加途径点、不放动态障碍，也不直接测试长路线。
4. 页面显示的位置只代表最后确认到达的网格节点，不代表节点之间的实时坐标。
5. 如果小车因故障或紧急停止在两个节点之间，禁止直接开始下一行程。必须关闭服务，把车放回可信节点中心，确认朝向后重新设置启动环境变量。

### 3.2 打开页面并验证初始化

树莓派执行以下命令查看局域网地址：

```bash
hostname -I
```

电脑和树莓派连接同一网络后，在电脑浏览器打开：

```text
http://<树莓派局域网地址>:8000/
```

不要把 `/docs` 当成用户前端；`/docs` 是 FastAPI 接口调试页面。

只打开页面，不提交行程，依次确认：

1. 页面可以显示 25 个网格点。
2. 页面小车位置与 `CAR_INITIAL_POSITION` 和实际摆放节点一致。
3. 页面没有显示“后端不可用”或“初始化失败”。
4. 小车电机不转，蜂鸣器保持静音。
5. 另一个 SSH 终端执行健康检查，结果仍为 `navigation_mode=hardware`、`hardware_ready=true`：

```bash
curl http://127.0.0.1:8000/api/health
curl http://127.0.0.1:8000/api/car/status
```

任何一项不一致都应停止，不要通过提交行程来试探问题。

### 3.3 测试单条直线边

假设小车实际位于 `A1`、朝东，并且 `A1 -> A2` 是已经通过手动导航验证的直线边：

1. 页面起点填写 `A1`，必须与小车当前位置一致。
2. 终点填写相邻节点 `A2`。
3. 不添加途径点，点击“叫车”一次。
4. 行程中起点、终点、途径点和叫车按钮应锁定，“重置”应变为“取消行程”。
5. 小车真实到达 `A2` 后，页面小车位置才能更新到 `A2`。

成功标准：

- 小车沿线到达 `A2` 并停车，没有持续驱动电机。
- 页面最终显示到达状态，路线进度到达 `A2`。
- 消息按实际进度出现，且同一条消息没有因轮询重复显示。
- “邮箱”页面显示到达记录；未配置真实邮件时状态应为 `disabled`，页面明确显示“当前未启用真实邮件发送”。

如果页面先显示 `A2`、小车随后才到，或者小车已到但页面长期不更新，都判定失败。

### 3.4 测试刷新恢复

单边测试通过后，重新摆车并重启服务，选择一条已手动验证、至少包含两条边的路线：

1. 提交行程后等待小车到达第一个中间节点。
2. 在小车继续执行时刷新浏览器页面一次，不要重复点击叫车。
3. 确认页面自动恢复当前活动行程、路线、已完成进度和消息。
4. 确认刷新页面没有创建第二个行程，小车最终只执行一次路线。

成功标准：刷新不会停车或重复发车，页面恢复后继续跟踪同一个行程 ID，最终位置与小车实际终点一致。

### 3.5 测试取消停车

刷新恢复通过后，重新摆车并重启服务，再选择一条至少包含两条边的已验证路线：

1. 点击“叫车”，确认小车已经开始运动。
2. 在到达终点前点击“取消行程”一次。
3. 确认页面显示“取消中”或“等待节点停车”，且小车不倒回已经经过的节点。
4. 观察小车继续完成当前前向边，在到达前方下一个节点后停车。
5. 确认页面先保持 `canceling`，节点停车完成后才变为 `canceled`，最终位置是实际停车节点。

成功标准：取消请求不会让小车停在边中间；小车在前方下一个可信节点停车且不恢复运动，页面不把取消显示成正常到达，也不生成新的到达邮件记录。

如果循迹失败、检测到障碍或执行安全回退，安全停车优先于“必须前进到下一节点”。这类情况按故障处理，关闭服务并人工确认小车可信位置后再测试。

完成每一阶段后，把实际结果追加到 `docs/test-records.md`，至少记录：

只有已经实机观察到的结果才能写入测试记录，不能根据页面状态推断硬件已经通过。

## 4. 当前实测导航基线

手动工具和网页实车模式使用同一组默认值：

```text
forward speed = 5
line turn speed = 80
left correction = 80
right correction = 100
search speed = 5
spin speed = 30
left pre-turn = 0.4 秒
right pre-turn = 0.3 秒
left U-turn pre-turn = 0.8 秒
turn acquire timeout = 5.0 秒
leave node minimum = 0.10 秒
node clear samples = 1
node confirm samples = 1
node center speed = 20
node center time = 0.10 秒
obstacle confirm samples = 2
line acquire timeout = 3.0 秒
line lost timeout = 5.0 秒
reverse speed = 5
reverse correction speed = 20
edge timeout = 20 秒
recovery timeout = 8 秒
ultrasonic threshold = 20 cm
loop delay = 0.02 秒
```

实车曾测得完整左转约 `0.6s`、右转约 `0.5s`、左掉头约 `1.2s`、右掉头约 `1.1s`；上表的 `0.4/0.3/0.8s` 是当前运行时的预转向时间，后续由传感器精细找线，两组数值不能混用。完整事实记录见 `docs/test-records.md`。
