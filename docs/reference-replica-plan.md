# 参考项目复刻开发计划与架构规范

本文档用于指导四人小组开发 Yahboom 4WD 树莓派智能小车项目。

目标不是照抄参考项目源码，而是尽量复刻参考项目的功能效果、演示路线和报告亮点，同时重新整理代码边界，避免把旧项目中的大文件、硬编码配置、敏感信息和不可控硬件逻辑直接搬进当前项目。

## 1. 当前目标

### 1.1 总目标

在当前 Yahboom 4WD 小车上，尽量复刻参考项目“单兵辅助侦察智能车”的主要能力，形成一个可演示、可解释、可维护的项目。

优先追求：

- 快速完成可运行演示。
- 四个人能并行开发。
- 代码不乱，模块边界清晰。
- 后续能继续加功能，但不提前做复杂工程化。
- 报告、PPT、演示视频能和源码相互对应。

### 1.2 本质需求

这个项目的核心不是“写很多功能”，而是让硬件、算法、图像、集成和文档形成稳定闭环：

1. 小车能安全运动。
2. 传感器数据可信。
3. 单个功能能独立测试。
4. 综合演示能按固定流程跑完。
5. 出问题时能定位到具体模块，而不是只能猜。

### 1.3 非目标

当前阶段不追求：

- ROS、微服务、完整 Web 后台等重工程化方案。
- 为未来假想功能设计插件系统。
- 直接复用参考项目中的整合入口文件。
- 复制参考项目中的邮箱授权码、API Key、微信密钥等敏感信息。
- 为同一份数据设计多个来源兜底。
- 为同一字段兼容多种键名。

## 2. 已知事实与关键假设

### 2.1 已知事实

- 当前硬件是 Yahboom 4WD 树莓派智能小车。
- 当前仓库主要包含项目说明、参考项目和参考项目模块分析，还没有正式业务源码。
- `参考项目/` 目录只作为迁移参考，默认只读。
- 参考源码存在硬编码配置、潜在敏感信息、GPIO 引脚不一定匹配、摄像头编号不一定匹配等风险。
- 参考项目功能很多，包括循迹、避障、A* 路径规划、邮件、人脸识别、全景拍摄、摩斯电码等。

### 2.2 当前假设

- 主要开发语言使用 Python。
- 不单独建设前端或 Java 后端，除非后续演示明确需要。
- 四个人都需要有相对独立的任务，不让所有人挤在一个大文件里。
- 项目最终以现场演示、报告和 PPT 为主要交付物。
- 代码先保证可读、可测、可解释，再考虑扩展。

如果后续老师要求必须包含 Web 控制台、数据库、AI 云接口等内容，应单独调整架构，不要强行塞进当前最小架构。

## 3. 复刻策略

### 3.1 复刻什么

复刻的是功能效果和演示价值：

- 小车运动控制。
- 循迹或路径行驶。
- 超声波避障。
- A* 路径规划。
- 摄像头拍照或图片传输。
- 颜色识别、二维码识别、人脸识别等视觉能力，按时间和硬件稳定性逐步添加。
- 邮件、蜂鸣器、LED、摩斯电码等展示增强功能。

### 3.2 不复刻什么

不复刻旧项目的混乱实现方式：

- 不直接运行 `SDC.py`、`All.py`、`Rescue.py`、`FIN.ipynb` 等整合入口。
- 不复制硬编码路径、IP、邮箱授权码、API Key。
- 不默认参考项目 GPIO 引脚和摄像头编号适用于当前小车。
- 不把所有功能继续写进一个大文件。
- 不保留错误历史逻辑的兼容层。

### 3.3 功能优先级

| 层级 | 功能 | 原因 |
| --- | --- | --- |
| P0 必做 | 电机控制、安全停车、GPIO 清理 | 所有演示的基础，风险最高 |
| P0 必做 | 循迹传感器读取、超声波测距 | 传感器可信后才能做自动行驶 |
| P0 必做 | A* 路径规划 | 纯算法，最容易先在本地验证 |
| P1 重要 | 循迹任务、避障任务、固定路线巡逻 | 构成核心演示主线 |
| P1 重要 | 摄像头拍照、图片保存或传输 | 贴合侦察车主题 |
| P2 加分 | 颜色识别、蜂鸣器、LED、摩斯电码 | 成本较低，展示效果明确 |
| P2 加分 | 邮件发送 | 需要配置隔离，不能写真实密钥 |
| P3 选做 | 人脸识别、二维码识别、全景拼接 | 依赖多，调试成本高 |
| P3 选做 | 语音、微信、云 AI 接口 | 网络、密钥、外设风险高 |

原则：先完成 P0 和 P1，再根据时间添加 P2/P3。不能为了追求“功能数量”牺牲小车安全和核心演示稳定性。

## 4. 推荐代码结构

建议从下面的轻量结构开始：

```text
4wd_project/
  src/
    main.py
    config.py
    hardware/
      motor.py
      line_sensor.py
      ultrasonic.py
      camera.py
      buzzer.py
      led.py
    algorithms/
      astar.py
      color_detect.py
      qr_detect.py
    tasks/
      line_follow.py
      obstacle_avoid.py
      patrol.py
      reconnaissance.py
    services/
      image_transfer.py
      mail_sender.py
    tools/
      test_motor.py
      test_line_sensor.py
      test_ultrasonic.py
      test_camera.py
  docs/
    reference-replica-plan.md
    pin_map.md
    runbook.md
    test-records.md
```

这是建议结构，不要求一次性全部创建。当前阶段可以按功能逐步增加文件。

### 4.1 为什么这样组织

这套结构的目标不是“看起来工程化”，而是把不同风险和不同开发节奏的代码隔离开：

| 层级 | 放什么 | 为什么这样放 |
| --- | --- | --- |
| `src/main.py` | 程序入口和运行模式选择 | 入口只负责调度，避免所有人把功能都写进一个大文件 |
| `src/config.py` | 已确认的 GPIO 引脚、速度、阈值、路径等配置 | 配置集中后，换引脚或调参数时不用到处找；也能明确哪些配置已经实测 |
| `src/hardware/` | 电机、循迹、超声波、摄像头、蜂鸣器、LED 等真实硬件读写 | 硬件代码风险最高，必须和算法、任务流程分开，方便单独测试和定位问题 |
| `src/algorithms/` | A*、颜色识别、二维码识别等算法 | 算法尽量不依赖 GPIO，能先在 Windows 本地验证，降低上车调试成本 |
| `src/tasks/` | 循迹、避障、巡逻、侦察等完整任务流程 | 任务层组合硬件和算法，表达“演示要做什么”，不直接关心底层引脚细节 |
| `src/services/` | 图片传输、邮件发送等外部通信能力 | 网络和账号配置容易失败，放到独立层，避免影响小车运动安全 |
| `src/tools/` | 手动测试脚本 | 每个脚本只验证一个硬件或小功能，方便分工、排错和记录验收结果 |
| `docs/` | 接线、运行、测试记录和开发计划 | 文档记录真实环境和验证结果，避免报告、PPT 和源码互相对不上 |

本质分层是：`hardware/` 解决“怎么和真实设备交互”，`algorithms/` 解决“怎么计算结果”，`tasks/` 解决“怎么把能力串成演示流程”，`services/` 解决“怎么和外部系统通信”。这样组织后，一个功能出问题时能更快判断是硬件问题、算法问题、流程问题还是网络配置问题。

### 4.2 建议文件职责

这些文件是“可能需要的文件清单”，不是一次性创建清单。当前只创建正在实现或正在测试的文件。

| 文件 | 功能 | 什么时候创建 |
| --- | --- | --- |
| `src/main.py` | 项目主入口，根据参数启动电机测试、循迹、避障、巡逻、侦察等模式 | 需要统一运行入口时创建 |
| `src/config.py` | 保存已确认的 GPIO 引脚、默认速度、传感器阈值、图片保存路径等非敏感配置 | 第一次出现多个模块共用配置时创建 |
| `src/hardware/motor.py` | 控制电机前进、后退、左转、右转、停止，并保证异常时停车 | 开始测试底盘运动前创建 |
| `src/hardware/line_sensor.py` | 读取循迹传感器状态，判断黑线和白底的原始输入 | 开始做循迹传感器只读测试前创建 |
| `src/hardware/ultrasonic.py` | 读取超声波距离，只负责测距，不直接控制电机 | 开始做避障前的距离测试时创建 |
| `src/hardware/camera.py` | 打开摄像头、拍照、保存图片，处理摄像头资源释放 | 开始做拍照或视觉识别前创建 |
| `src/hardware/buzzer.py` | 控制蜂鸣器发声，用于状态提示或摩斯电码展示 | 需要声音提示或加分展示时创建 |
| `src/hardware/led.py` | 控制 LED 亮灭或闪烁，用于状态提示 | 需要灯光状态提示时创建 |
| `src/algorithms/astar.py` | 实现 A* 路径规划，输入地图、起点、终点，输出路径 | 做路径规划本地验证时创建 |
| `src/algorithms/color_detect.py` | 识别图片或摄像头画面中的目标颜色，输出颜色位置或识别结果 | 摄像头拍照稳定后再创建 |
| `src/algorithms/qr_detect.py` | 识别二维码内容，输出解码文本或识别失败原因 | 需要二维码演示且摄像头可用时创建 |
| `src/tasks/line_follow.py` | 组合循迹传感器和电机，实现沿线行驶流程 | 循迹传感器和电机都单独验证后创建 |
| `src/tasks/obstacle_avoid.py` | 组合超声波和电机，实现停车、绕行或避障策略 | 超声波测距和电机控制都稳定后创建 |
| `src/tasks/patrol.py` | 组合多个基础动作，执行固定路线巡逻 | 需要主演示流程时创建 |
| `src/tasks/reconnaissance.py` | 组合巡逻、停车、拍照、识别、传输等侦察流程 | 单功能都验证后再创建 |
| `src/services/image_transfer.py` | 发送或保存侦察图片，例如传到电脑、局域网服务或指定目录 | 图片采集稳定且需要展示传输时创建 |
| `src/services/mail_sender.py` | 发送邮件通知或图片附件，账号和授权码必须从本地环境读取 | 确认要演示邮件功能时创建 |
| `src/tools/test_motor.py` | 手动验证电机低速短时运动和停止逻辑 | `motor.py` 创建后立刻创建 |
| `src/tools/test_line_sensor.py` | 手动打印循迹传感器读数，验证黑线和白底区别 | `line_sensor.py` 创建后立刻创建 |
| `src/tools/test_ultrasonic.py` | 手动连续打印超声波距离，不控制电机 | `ultrasonic.py` 创建后立刻创建 |
| `src/tools/test_camera.py` | 手动拍一张照片并保存，验证摄像头是否可用 | `camera.py` 创建后立刻创建 |
| `docs/reference-replica-plan.md` | 记录复刻目标、架构边界、开发计划和验收标准 | 已创建，作为小组协作总纲 |
| `docs/pin_map.md` | 记录当前小车真实接线、GPIO 引脚来源和实测结果 | 写任何 GPIO 代码前优先创建 |
| `docs/runbook.md` | 记录上传、运行、停止、常见故障处理步骤 | 开始在树莓派运行代码前创建 |
| `docs/test-records.md` | 记录每次硬件和功能测试的日期、现象、结论和问题 | 开始真实硬件测试时创建 |

最小起步建议：先做 `config.py`、`hardware/motor.py`、`hardware/ultrasonic.py`、`hardware/line_sensor.py`、`algorithms/astar.py` 和对应 `tools/test_*.py`。这些文件覆盖底盘安全、基础传感器和纯算法，是后续综合演示的最低可信基础。

## 5. 模块边界规范

### 5.1 `main.py`

职责：

- 作为程序入口。
- 根据参数选择运行模式。
- 调用 `tasks/` 中的任务。

禁止：

- 直接写 GPIO 控制细节。
- 直接写摄像头识别细节。
- 直接堆 if/else 实现所有业务。

### 5.2 `config.py`

职责：

- 存放 GPIO 引脚、默认速度、阈值、路径等配置。
- 只放当前硬件已确认的配置。

禁止：

- 存放密码、授权码、API Key。
- 同一个配置保留多个备用字段。
- 为未确认硬件写大量兜底配置。

### 5.3 `hardware/`

职责：

- 封装真实硬件读写。
- 电机、传感器、摄像头、蜂鸣器、LED 分文件维护。

边界：

- 可以 import `RPi.GPIO`、`cv2` 等硬件相关库。
- 不写复杂业务流程。
- 所有可能驱动车轮的代码必须有停止逻辑。
- GPIO 程序必须保证异常时执行 `GPIO.cleanup()`。

### 5.4 `algorithms/`

职责：

- 放纯算法和图像算法。
- 输入明确，输出明确，尽量能在 Windows 本地测试。

边界：

- 不 import 电机模块。
- 不直接操作 GPIO。
- 不依赖树莓派真实硬件。

### 5.5 `tasks/`

职责：

- 组合硬件和算法，形成可演示任务。
- 例如循迹任务、避障任务、巡逻任务、侦察任务。

边界：

- 可以调用 `hardware/` 和 `algorithms/`。
- 负责流程控制和状态切换。
- 不直接写底层 GPIO 细节。

### 5.6 `services/`

职责：

- 放外部通信能力，例如图片传输、邮件发送。

边界：

- 密钥和账号从环境变量或本地不提交配置读取。
- 网络失败应明确报错，不能静默吞掉。
- 不影响核心行驶安全。

### 5.7 `tools/`

职责：

- 放手动测试脚本。
- 每个脚本只测一个硬件或一个小功能。

示例：

- `test_motor.py`：低速前进 0.5 秒后停止。
- `test_ultrasonic.py`：持续打印距离，不控制电机。
- `test_camera.py`：拍一张图片保存到本地。

## 6. 四人分工

### 成员 A：底盘与安全

负责：

- 电机控制。
- GPIO 初始化和清理。
- 安全停车。
- 电机测试脚本。

交付标准：

- 小车能低速短时运动。
- 程序正常结束、异常退出、手动中断时都能停止。
- 电机模块接口简单，例如前进、后退、左转、右转、停止。

### 成员 B：传感器与视觉输入

负责：

- 循迹传感器读取。
- 超声波测距。
- 摄像头拍照。
- 颜色识别、二维码识别等视觉输入能力。

交付标准：

- 每个传感器先有只读测试脚本。
- 数据能稳定打印或保存。
- 不在传感器测试阶段直接控制电机。

### 成员 C：算法与任务策略

负责：

- A* 路径规划。
- 循迹策略。
- 避障策略。
- 任务状态机。

交付标准：

- A* 能在本地固定地图上跑通。
- 策略逻辑尽量不依赖真实 GPIO。
- 任务流程能被主入口调用。

### 成员 D：集成、文档与演示

负责：

- `main.py` 运行入口。
- 演示模式整合。
- 项目运行说明。
- 测试记录。
- 报告和 PPT 的技术内容对齐。

交付标准：

- 能说明每个模块怎么运行。
- 能记录哪些功能完成、哪些功能未完成、限制是什么。
- 最终演示流程固定，不临场拼命改代码。

## 7. 阶段开发计划

### 阶段 0：硬件与资料确认

目标：

- 确认当前小车实际接线、摄像头、传感器和运行目录。
- 明确参考项目哪些功能要复刻，哪些只是加分。

动作：

- 建立 `docs/pin_map.md`。
- 建立 `docs/runbook.md`。
- 从参考项目模块分析中挑选可迁移功能。

验证：

- 每个 GPIO 引脚都有来源说明。
- 运行步骤能让组内成员照着执行。
- 没有把敏感信息写进文档或代码。

### 阶段 1：硬件最小闭环

目标：

- 先让小车安全地动起来。
- 先读传感器，不急着自动驾驶。

动作：

- 实现电机控制。
- 实现超声波只读测距。
- 实现循迹传感器只读读取。
- 实现摄像头拍照测试。

验证：

- 电机低速运行固定时间后必停。
- 手动中断后小车必停。
- 超声波能连续输出合理距离。
- 循迹传感器能区分黑线和白底。
- 摄像头能保存一张可查看图片。

### 阶段 2：单功能复刻

目标：

- 把参考项目中的主要单功能拆开重写。

动作：

- 实现循迹任务。
- 实现避障任务。
- 实现 A* 路径规划。
- 实现图片保存或传输。
- 根据时间添加颜色识别、蜂鸣器、LED、摩斯电码。

验证：

- 每个功能都有单独入口或测试脚本。
- 单功能失败不影响其它模块调试。
- A* 路径规划能用固定地图输出正确路径。
- 视觉功能能保存结果图片或打印识别结果。

### 阶段 3：综合演示流程

目标：

- 形成一个稳定演示路线，尽量体现参考项目的综合能力。

建议演示流程：

1. 启动主程序。
2. 小车执行固定巡逻或循迹。
3. 遇到障碍物时停车或绕行。
4. 到达指定位置后拍照。
5. 可选：进行颜色识别、二维码识别或人脸识别。
6. 可选：发送图片或邮件。
7. 可选：通过蜂鸣器或 LED 给出状态提示。

验证：

- 演示流程能连续跑完。
- 中途异常时小车停止。
- 每个展示点能解释对应代码模块。
- 报告和 PPT 中的描述与实际源码一致。

### 阶段 4：高级功能补齐

目标：

- 在核心演示稳定后，尽量补齐参考项目中的高级功能。

候选功能：

- 人脸识别。
- 二维码识别。
- 全景拍摄和图像拼接。
- 邮件发送。
- 云 API 或 AI 接口。
- 语音或微信消息。

验证：

- 每个高级功能都能独立开关。
- 缺少网络、密钥或外设时，不影响核心行驶演示。
- 所有敏感配置不提交到仓库。

## 8. 开发规范

### 8.1 简单优先

- 用最少代码解决当前真实问题。
- 不为一次性代码建立抽象。
- 不为假想场景添加配置项。
- 不写只转发一行调用的包装函数。

### 8.2 文件大小控制

如果一个文件同时做了硬件控制、算法、网络、演示流程，说明边界已经错误。

建议：

- 一个硬件一个文件。
- 一个任务一个文件。
- 一个算法一个文件。
- 主入口只负责调度。

### 8.3 配置和敏感信息

禁止提交：

- 邮箱授权码。
- API Key。
- 微信密钥。
- 个人账号密码。
- 临时公网地址。

允许提交：

- `.env.example`。
- 配置字段名。
- 示例占位值。

### 8.4 错误处理

硬件层：

- 电机相关异常必须尽快停车。
- GPIO 程序必须清理资源。

算法层：

- 输入不符合契约时应尽早报错。
- 不为同一字段兼容多个名字。

服务层：

- 网络失败应打印明确错误。
- 不静默吞异常。

### 8.5 参考项目迁移规则

迁移前先回答：

1. 这个功能的输入是什么？
2. 输出是什么？
3. 是否依赖 GPIO、摄像头、网络或密钥？
4. 是否能拆成独立模块？
5. 当前小车硬件是否支持？

迁移方式：

- 可以参考旧代码的算法思想。
- 可以参考旧报告的功能描述。
- 不能直接复制整合入口。
- 不能复制敏感配置。
- 不能默认旧 GPIO 引脚正确。

## 9. 测试与验收标准

### 9.1 单模块验收

| 模块 | 验收标准 |
| --- | --- |
| 电机 | 低速短时运动后必停，中断后必停 |
| 超声波 | 连续输出合理距离，异常值可观察 |
| 循迹 | 能区分黑线和白底 |
| 摄像头 | 能保存清晰图片 |
| A* | 固定地图输入能输出预期路径 |
| 邮件 | 使用占位配置或本地环境变量，不提交真实密钥 |
| 图片传输 | 发送端和接收端协议一致 |
| 颜色识别 | 能在现场光照下识别目标颜色 |

### 9.2 综合验收

项目可以认为达到可交付状态，需要满足：

- 有一个稳定主演示流程。
- 主演示流程能在现场重复运行。
- 小车异常时能停。
- 每个核心功能都有独立测试方式。
- 文档能说明如何运行、如何接线、如何复现演示。
- 报告中没有宣称源码没有实现的能力。

## 10. 主要风险

| 风险 | 影响 | 处理方式 |
| --- | --- | --- |
| GPIO 引脚不匹配 | 小车不动或行为异常 | 先建立接线表，只按实测配置写代码 |
| 电机失控 | 安全风险高 | 所有运动测试低速短时，必须有停止逻辑 |
| 摄像头被占用 | 拍照失败 | 先单独测试摄像头，再集成 |
| 参考源码包含敏感信息 | 泄露风险 | 不复制密钥，使用 `.env.example` |
| 高级功能依赖网络 | 演示不稳定 | 高级功能放在核心演示之后 |
| 所有人改同一个大文件 | 冲突多、难维护 | 按模块分文件，主入口只调度 |
| 报告和源码不一致 | 答辩风险 | 文档和 PPT 只写已经验证的功能 |

## 11. 下一步建议

建议按下面顺序推进：

1. 创建 `docs/pin_map.md`，记录当前小车真实接线。
2. 创建 `docs/runbook.md`，记录树莓派上传、运行和停止方式。
3. 创建 `src/` 基础目录，但只先实现电机、传感器、A* 三类最小模块。
4. 每个人领取一个明确模块，先写独立测试脚本。
5. 核心功能稳定后，再按优先级补齐参考项目中的高级功能。

最重要的判断标准：如果一个功能不能单独测试，就先不要接入综合演示。
