# 网格导航代码地图实施问题记录

本文件按 `docs/交互式代码地图生成标准流程.md` 记录实施中真实发生、已经验证且可复用的问题。

### 方法 ID 与步骤 ID 发生跨类型重名

- 问题现象：首次静态检查发现 `astar-neighbors`、`radar-pattern`、`sensor-normalize` 同时被方法和步骤使用，不满足 SOP 要求的全局 ID 唯一性；继续保留会让数据引用排查变得含混。
- 原因：方法 ID 按功能命名后，三个调用步骤沿用了同一个自然名称，没有在写入时区分“方法节点”和“方法内步骤”。
- 解决方案：保留稳定的方法 ID，把三个步骤分别改为 `astar-neighbors-step`、`radar-pattern-step`、`sensor-normalize-step`，并同步更新对应调用的 `sourceStepId`。
- 验证结果：重新运行静态检查后，4 个目录、8 个文件、53 个方法、132 条调用及全部步骤的 ID 均唯一；调用引用、源码行号、参数和入口可达性同时通过。
- 可复用经验：生成地图数据时应从一开始让步骤 ID 带 `-step` 或明确的动作前缀，避免与方法、文件等节点 ID 共享自然名称。

### 模板的不同文件调用线默认假设文件横向分列

- 问题现象：真实浏览器几何检查发现，`tasks/` 目录内 `grid_navigation.py`、`edge_follow.py`、`line_follow.py`、`reverse_radar.py` 是纵向堆叠的；模板原先对“不同文件”统一取两个文件水平边界的中点，导致同列文件调用线以及偏移过大的跨目录调用线穿过方法正文。
- 原因：模板示例的三个文件分别位于三个目录列，没有覆盖“同一目录列中多个文件上下排列”的路由场景；仅调整少量 lane 无法改变同列文件的错误基准通道。
- 解决方案：在本地图复制件的 `call-lines.js` 中区分三种位置关系：同一文件使用文件内右侧预留区、左右分列文件使用真实列间中点、水平范围重叠的纵向文件统一绕到整列右侧；同时把跨文件 lane 收敛到 `-5` 至 `5` 的安全范围。
- 验证结果：重新构建后，入口单独展开时 31 条线、入口与 `execute_planned_edge()` 同时展开时 40 条线均保留；浏览器逐条检查垂直线段与所有方法正文矩形的交集，结果为 0，节点重叠和文字溢出也均为 0。
- 可复用经验：复制模板后如果一个目录中有多个纵向文件，不能只看目录数量是否匹配；还要按文件矩形的水平关系选择路由基准，并在至少两个来源方法同时展开时检查连线是否穿过正文。

本次没有仍未解决的问题。
